# Jarvis MVP — монитор сессий Claude Code

Меню-бар приложение для macOS на **Rust + Tauri 2**: видит все сессии Claude
Code во всех терминалах и говорит, когда какая-то **закончила ответ** или
**ждёт тебя** (пермишен/ввод).

## Как это работает

```
claude (любой терминал)
  └─ hooks из ~/.claude/settings.json   ← вшивает npm run setup
       └─ ~/.jarvis/bin/jarvis-hook     ← fail-silent шим, curl за 0.3с
            └─ unix-сокет ~/.jarvis/run.sock
                 └─ Rust-демон (Tauri) = реестр сессий
                      ├─ собственные тосты-уведомления (поверх фуллскрина)
                      ├─ панель (alwaysOnTop, поверх фуллскрина, фокус не ворует)
                      └─ счётчик в меню-баре: ⏸ ждут · ⚙ работают
```

События: `SessionStart` → idle, `UserPromptSubmit` → working,
`Notification` → waiting (+уведомление), `Stop` → done (+уведомление),
`SessionEnd` → сессия исчезает. Никакого парсинга экрана — только
структурированные события от самого Claude Code (экран паны читается только
детектором интерактивных промптов и пультом).

Архитектура: фронтенд (панель `ui/index.html` + `ui/renderer.js`,
тосты `ui/toast.*`) работает в системном WKWebView; контракт `window.jarvis`
реализует тонкий адаптер `ui/bridge.js` поверх Tauri IPC. Main-процесс на Rust
(`src-tauri/src/`): unix-сокет на axum, реестр-редьюсер с эффектами, tmux,
транскрипты, usage/история, haiku-вызовы, плагины питания (IOPMAssertion
через IOKit — краш демона автоматически снимает запрет сна).
Состояние, настройки и статистика лежат в `~/.jarvis/` в стабильных форматах.

## Быстрый старт

Нужны: Rust 1.88+ (`brew install rust` / rustup), tmux (опционально, для
пульта и ответов).

```bash
npm run setup     # вшить хуки в ~/.claude/settings.json (с бэкапом, идемпотентно)
npm start         # собрать и поднять демон + меню-бар (◇ справа сверху)
```

(npm здесь — только привычный раннер: внутри это `cargo run --release`.
Без npm: `cargo run --release --manifest-path src-tauri/Cargo.toml`.)

Дальше **перезапусти активные сессии** `claude` — хуки снимаются снапшотом
на старте сессии. Если Claude Code попросит подтвердить изменённые хуки
(меню `/hooks`) — это наша запись, `~/.jarvis/bin/jarvis-hook`.

Клик по ◇ — панель. Правый клик — меню (тест уведомления, автозапуск, выход).

Глобальный хоткей **⌘J** открывает панель по центру экрана (как Raycast):
Esc, клик мимо или повторный ⌘J — закрыть. Хоткей, позиция панели,
тумблеры уведомлений и автозапуск — в настройках (⚙ в шапке панели),
хранятся в `~/.jarvis/settings.json`. Учти: глобальный ⌘J перехватывает
эту клавишу во всех приложениях (Chrome «загрузки», VS Code «панель») —
поменяй сочетание в настройках, если мешает.

## Пощупать без Claude Code

Вручную, как это делает настоящий хук (демон должен быть запущен):

```bash
echo '{"session_id":"t1","cwd":"'$PWD'"}' | ~/.jarvis/bin/jarvis-hook claude session-start
echo '{"session_id":"t1","cwd":"'$PWD'","prompt":"добавь тесты"}' | ~/.jarvis/bin/jarvis-hook claude prompt
echo '{"session_id":"t1","cwd":"'$PWD'","message":"Claude needs your permission to use Bash"}' | ~/.jarvis/bin/jarvis-hook claude notification
echo '{"session_id":"t1","cwd":"'$PWD'"}' | ~/.jarvis/bin/jarvis-hook claude stop
```

Проверка, что демон жив: `curl -s --unix-socket ~/.jarvis/run.sock http://jarvis/`

## tmux-транспорт и «Ответить»

`npm run setup` ставит PATH-шим `~/.jarvis/shims/claude` (паттерн pyenv,
блок в `~/.zshrc` между `# >>> jarvis >>>` / `# <<< jarvis <<<`). После
`exec zsh` каждый интерактивный запуск `claude` прозрачно оборачивается
в tmux на **отдельном сервере** (`-L jarvis`, конфиг `~/.jarvis/tmux.conf`):
без статус-бара, мышь скроллит, личный tmux не затрагивается. В iTerm2 —
control mode (`-CC`), родные табы. Headless-запуски (`-p`, пайпы, `$TMUX`)
не оборачиваются.

Зачем: у демона появляется канал **ввода**. В чате сессии (Enter на строке)
поле «Ответить» вставляет текст прямо в терминал сессии — даже если окно
свёрнуто или на другом Space. Вставка: `set-buffer → paste-buffer -p →
send-keys Enter` (многострочные промпты доходят одним куском).

- Сессия вне tmux (запуск в обход шима, headless) — помечена «вне tmux»:
  управлять ей из Jarvis нельзя (ни ответ, ни пульт). Панель показывает, что
  запустить, чтобы завести её в tmux: `claude --resume <session_id>` — shim
  завернёт продолжение в наш tmux-сервер, и управление заработает.
- Закрыл окно терминала — агент жив: tmux-сессия уходит в detach, Jarvis
  продолжает следить. Переподключиться: `tmux -L jarvis attach -t <имя>`
  (имена видны в `tmux -L jarvis ls` и совпадают с проектом в панели).
- `npm run teardown` убирает шим, конфиг и PATH-блок; живые tmux-сессии
  не убивает (предупредит). `npm run status` показывает каждый компонент.

## Пульт: модель и effort

В чате tmux-сессии под полем ответа — пульт: сегменты **Модель**
(Opus / Sonnet / Haiku) и **Effort** (auto / low / med / high / xhigh).
Клик шлёт слэш-команду с аргументом (`/model sonnet`, `/effort high`) —
форма без слайдера, исполняется в живой сессии сразу. Каналы те же, что у
ответа и палитры команд.

- **Модель** видна бесплатно — пишется в transcript в каждом ходе ассистента
  (`message.model`); бейдж в строке сессии показывает её без запросов. Если
  Claude Code форкнул сессию в новый файл, модель читается из самого свежего
  транскрипта каталога проекта.
- **Effort** снаружи не читается (нет ни в transcript, ни в `/status`),
  поэтому панель ведёт оптимистичное состояние — подсвечивает то, что сама
  выставила.
- Управление — только для tmux-сессий. Палитра команд (`/` в поле ответа)
  даёт доступ к остальным слэш-командам той же сессии.

## Плагины питания: «Не спать» (☕) и «Крышка» (⌒)

Уснувший мак — замороженные claude-процессы и оборванные API-запросы.
Оба режима — **подключаемые плагины** (тумблеры в настройках панели),
UX списан с Raycast Coffee и Amphetamine.

**☕ Не спать** — вето на idle-сон через power assertion (IOPMAssertion из
IOKit, те же ассершены, что у caffeinate). Как у Caffeine/Amphetamine:
ничего не делает, пока сам не включишь — правый клик по ◇ → секция «Не спать»:

- **Бессрочно** — пока не выключишь.
- **На время** — пресеты 15м…8ч (семантика `caffeinate -t`).
- **Пока жив процесс** — claude-сессии Jarvis и GUI-приложения
  (семантика `caffeinate -w`).
- **Пока агенты работают (авто)** — опциональный тумблер (выкл по
  умолчанию): assertion берётся при первой working-сессии и отпускается
  через минуту после того, как все затихли. Экран гаснет, мак работает.

Ручной режим один: новый старт заменяет предыдущий; авто-триггер,
если включён, живёт независимо от ручного.

Assertion живёт в процессе демона: краш = автоснятие, «застрявший» запрет
сна невозможен. Проверка: `pmset -g assertions | grep -i jarvis`.

**⌒ Крышка** — closed-display mode: мак работает даже с закрытой крышкой
(`pmset -a disablesleep`, уровень IOPMrootDomain). Это root и термо-риски,
поэтому плагин **детектит и подсказывает, а не молча sudo**:

- Проснулся после сна, который прервал работающие сессии → уведомление
  «включить closed-display?» (с внешним дисплеем посоветует родной
  clamshell-режим — там root не нужен). Сон детектится по разрыву
  секундного тика демона — без приватных API.
- Ручной тумблер в трее — честный admin-диалог. Для тихого переключения
  (и режима «авто при работе агентов») — «Настроить тихий режим»:
  ставит `/etc/sudoers.d/jarvis-pmset` (ровно две команды, через `visudo -c`).
- Fail-safe как у Amphetamine Enhancer: маркер `~/.jarvis/clamshell.json`,
  восстановление сна на старте/выходе демона, батарейный сторож
  (≤15% на батарее → вернуть сон, в безвыходном случае `pmset sleepnow`).
- MacBook Air без вентилятора под крышкой троттлит — плагин предупреждает.

## Доска задач (наблюдатель, не владелец)

Сессии, которые гоняют многошаговый план (оркестратор ведёт чек-лист через
`TodoWrite` и спавнит сабагентов), показывают в чате кнопку-кольцо «N/M задач».
Клик → слайд-овер «Задачи»: агрегат «выполнено · в работе · в очереди»,
прогресс-бар, список задач со статусом (готово / в работе / в очереди /
прервано), где удаётся — модель и время скоррелированного сабагента. Доска
живая: следующий `TodoWrite` перерисовывает её сам, без действий пользователя.
Сессии без плана доски не показывают.

**Граница: Jarvis — наблюдатель с грубым пультом, а не владелец плана.**

- Источник истины по задачам — **оркестратор внутри сессии** (встроенный
  goal-mode, superpowers, любой). У списка `TodoWrite` нет внешнего API «отметь
  задачу сделанной» — мутировать его извне нельзя.
- Jarvis доску **только читает** (событие `TodoWrite` через `PostToolUse`,
  last-write-wins: каждый вызов несёт полный список и заменяет предыдущий) и
  **отображает**. Кнопки доски её **не двигают**.
- Действие с задачи (перейти / пропустить / перезапустить) — это **инструкция
  оркестратору, а не мутация**: оно префилит поле ответа редактируемым текстом
  («Перейди к задаче N сейчас…»), который ты правишь и отправляешь сам обычным
  каналом `Ответить` (tmux send-keys). Доска **не меняется по нажатию** — только
  когда прилетит следующий реальный `TodoWrite`. Если агент проигнорировал
  инструкцию, доска покажет, что он реально делает, а не что мы попросили.
- Сабагенты ведутся отдельным реестром (старт — `PreToolUse(Task)`, стоп —
  `PostToolUse(Task)`; `SubagentStart` как хук в Claude Code отсутствует).
  Привязка сабагента к задаче — эвристика по «Task N» в описании; нет уверенного
  совпадения → сабагент идёт в отдельную полоску, без выдуманной привязки.
- Сценарий «Jarvis сам держит план и дёргает claude/codex как воркеров» — это
  **отдельная оркестратор-фаза, вне этого инкремента**. Здесь ни одна кнопка не
  ведёт себя так, будто Jarvis владеет планом.

## Голос (TTS) — Jarvis говорит

После события сессии Jarvis коротко проговаривает по-русски, что произошло,
локальным TTS. Текст собирается **шаблоном из структурных сигналов** (task-доска
> diff-статы > голый факт), числа разворачиваются в слова с русским
согласованием («одна задача» / «две задачи» / «пять задач», «сброс через два
часа четырнадцать минут»):

- ход закончен → «Пиксела: четыре из шести задач, сейчас docker-compose» (есть
  доска) · «Рекрю готов, изменено три файла» (есть diff) · «Тикетинг закончил»;
- ждёт тебя (пермишен/ввод) → **приоритетно**: «Пиксела ждёт — нужно разрешение
  на Bash»; упёрлась в лимит → «Пиксела упёрлась в лимит, сброс через два часа».

Реплики **не накладываются**: очередь сериализована, «ждёт» идёт вперёд
«готово», повтор idle-уведомления не озвучивается дважды, а затор `Stop`-реплик
коалесцируется («Пиксела и Рекрю закончили»).

**Движок — на выбор, флагом в конфиге** `~/.jarvis/settings.json`:

```json
"voice": { "engine": "piper", "mute": false,
           "events": { "stop": true, "notification": true, "stopFailure": true } }
```

Сменил `engine` и перезапустил демон — голос меняется **без правок кода**. Два
движка за единым трейтом `TtsEngine`:

- **Piper** (готов) — лёгкий, быстрый: бинарь + ~десятки МБ модели, ставится
  установщиком в `~/.jarvis/`. Минус — слабее на русском и заметно коверкает
  английские вставки.
- **Silero** (Фаза 2, пока заглушка) — лучший русский и смешанная речь, но
  тяжёлый Python-сайдкар (PyTorch, сотни МБ–ГБ) и медленнее старт. При
  `engine="silero"` сейчас демон молчит с внятной причиной в логе — это
  безопасный fail-safe до второй фазы.

Меню-бар: тумблер **«Без звука»** (мгновенно глушит очередь; баннеры-уведомления
остаются) и **«Тест голоса»** (образец текущим движком). Установка Piper —
`npm run setup`; проверить состояние движков — `npm run status`.

**Граница: это вывод-only — «Jarvis говорит».** Голосовой ввод (микрофон,
wake-word, STT, ответ агенту голосом) — **следующая фаза, отдельная спека.**
Если сбой/отсутствие движка — демон работает как раньше, Claude Code не затронут,
причина в логе.

## Что куда пишется

- `~/.jarvis/bin/jarvis-hook` — шим (вся логика в приложении, запись в конфиге не меняется никогда)
- `~/.claude/settings.json` — добавляются записи в `hooks` (merge, не overwrite; бэкап рядом: `settings.json.bak-...`)
- `~/.jarvis/run.sock` — сокет демона
- `~/.jarvis/shims/claude` — PATH-шим tmux-обёртки
- `~/.jarvis/tmux.conf` — конфиг отдельного tmux-сервера (`-L jarvis`)
- `~/.zshrc` — managed-блок PATH между маркерами jarvis (merge, бэкап, atomic)
- `~/.jarvis/settings.json` — настройки приложения (хоткей, уведомления, позиция, `plugins.*`)
- `~/.jarvis/state.json` — реестр сессий (переживает перезапуск демона)
- `~/.jarvis/usage.json` / `history.json` / `translations.json` — кэши статистики, истории, переводов
- `~/.jarvis/clamshell.json` — маркер поднятого disablesleep (fail-safe «Крышки»)
- `/etc/sudoers.d/jarvis-pmset` — опционально, по явной установке: тихий `pmset disablesleep`

Снести всё: `npm run teardown`. Посмотреть состояние: `npm run status`.

## Известные ограничения MVP

- **Схема хуков Claude Code дрейфует между версиями.** Текущая запись —
  под схему `hooks.<Event>[].hooks[].command`. Если после апдейта `claude`
  события перестали приходить — сверь с актуальной докой по hooks и поправь
  `EVENTS`/формат в `src-tauri/src/bin/setup.rs`.
- Иконка в меню-баре текстовая (◇ + счётчики) — нормальная template-иконка
  позже.
- Если убить терминал жёстко (без `SessionEnd`), сессия останется висеть —
  кнопка «Очистить» в панели убирает done/idle.
- Меню трея собирается заранее, а не в момент клика: список «Пока жив
  процесс» и строка состояния крышки освежаются на правый клик и доезжают
  к следующему открытию меню.
- Codex, голос, TTS-саммари — сознательно за бортом этой итерации.
