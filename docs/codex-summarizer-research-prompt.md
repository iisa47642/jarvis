# Промт для Codex: ресёрч по ускорению суммаризатора Jarvis

> Скопируй блок ниже целиком в Codex. Промт самодостаточный — доступ к
> запущенному приложению и логам не нужен.

---

Ты — инженер-ресёрчер. Задача: предложить, как ускорить «суммаризатор» в проекте
Jarvis. У тебя НЕТ доступа к запущенному приложению и логам — работай по фактам
ниже + ищи в интернете (модели, бенчмарки, подходы). Нужен разбор вариантов и
конкретная рекомендация, а не код.

## Что такое Jarvis и при чём тут суммаризатор
Jarvis — macOS-приложение (демон на Rust + Tauri 2, Apple Silicon), монитор сессий
Claude Code. Когда агент в сессии заканчивает ход (событие Stop), Jarvis должен
показать всплывающее уведомление-тост с короткой русской выжимкой «что по сути
сделано» (1 предложение, до ~160 символов) и озвучить её локальным TTS (Silero).

Сейчас выжимку делает «суммаризатор»: демон берёт хвост финального ответа агента
(до 3000 символов) и зовёт модель, прося короткое русское предложение. Реализовано
так: демон запускает **CLI `claude`** как подпроцесс с промптом (модель haiku),
ждёт ответ, парсит. Промпт просит СТРОГО JSON:
  {"display": "<текст для экрана, англ. термины можно латиницей>",
   "speak":   "<тот же смысл, но как ПРОИЗНОСИТЬ вслух по-русски: англ. слова
               транслитерировать кириллицей (GitLab→гитлаб), числа словами>"}
`display` идёт в тост/список, `speak` — в TTS.

ВАЖНО по архитектуре: тост показывается ТОЛЬКО после готовой выжимки (ждём
суммаризатор), поэтому задержка уведомления == время суммаризатора.

## Замеренная проблема (факт)
Тайминг «конец ответа агента → показано уведомление»: **≈14.9 секунды**, из них
**≈14.93с — это сам вызов haiku** (99.97% времени). То есть бутылочное горлышко —
суммаризатор. 15 секунд на одно короткое саммари — это очень много. Вероятные
причины (проверь/оцени): холодный старт/оверхед запуска CLI `claude` на каждый
вызов; маршрутизация HTTP через корпоративный прокси (он же тормозит другой
трафик); отсутствие стриминга; большой входной промпт.

## Цель
Уменьшить задержку «результат → уведомление» до **<2с** (идеально <1с), сохранив
качество: одно осмысленное русское предложение + русско-произносимая версия.
Пользователь ЯВНО заинтересован в варианте с **локальной моделью** (скачать один
раз, работает офлайн, НЕ тратит API-токены) — для него это даже предпочтительнее.

## Ограничения и контекст рантайма
- Хост: Mac на Apple Silicon (M-серия), обычно 16–32 ГБ RAM.
- Рядом уже крутится Python-сайдкар Silero TTS (PyTorch, в покое ~250 МБ RAM,
  на синтезе пик ~900 МБ, ~1 ядро CPU). Локальный суммаризатор добавится сверху —
  хочется лёгкий: маленький RAM/CPU-след, быстрый инференс на CPU/MPS.
- В проекте УЖЕ есть паттерн «Python-сайдкар, которым рулит демон» (Silero:
  FastAPI на localhost, демон шлёт текст, получает результат, супервизор
  перезапускает). Локальную LLM можно встроить тем же способом.
- Вход: до 3000 символов смешанного RU/EN текста (дев-диалог: код, команды,
  английские термины). Выход: 1 короткое русское предложение + speak-версия.
- Суммаризатор в коде — заменяемый компонент с чистым интерфейсом
  «текст ответа → строка(и)», так что замена движка не трогает остальной код.

## Что нужно от тебя (ресёрч + рекомендация)
1. **Диагностика текущего пути**: почему вызов haiku через CLI может занимать ~15с
   и как это срезать БЕЗ смены модели (убрать прокси для этого вызова; держать
   тёплый процесс/демон вместо запуска CLI на каждый раз; стриминг; урезать вход;
   кэш; параллельность). Оцени реалистичный потолок такой оптимизации.
2. **Локальная альтернатива (главное)**: подбери конкретные локальные модели-
   суммаризаторы под эту задачу (короткое RU-саммари дев-текста), которые:
   - запускаются на Apple Silicon офлайн, быстро (цель: первый токен <0.5с,
     полное саммари <2с на CPU/MPS), с малым RAM (желательно <2–3 ГБ);
   - хорошо работают с РУССКИМ и смешанным RU/EN текстом.
   Рассмотри классы: квантованные малые LLM через llama.cpp/Ollama/MLX (например
   Qwen2.5-1.5B/3B-Instruct, Gemma-2-2B, Phi-3.5-mini, Mistral-7B-Q4, Saiga/
   ru-адаптации, Vikhr и т.п.); seq2seq-суммаризаторы (mT5/ruT5/FRED-T5);
   экстрактивные/гибридные подходы. Дай таблицу: модель — размер — RAM — скорость
   на M-Mac — качество русского — лицензия — как запускать (llama.cpp/Ollama/MLX/
   transformers). Учти, что вывод нужен СТРОГО структурированный (JSON display/
   speak) — оцени, какие модели надёжно дают JSON по инструкции малого размера.
3. **Можно ли вообще без LLM**: достаточно ли качественной шаблонной/экстрактивной
   выжимки + правил (для скорости и нуля токенов)? Где грань.
4. **Архитектурное**: стоит ли показывать тост сразу (мгновенно, базовым текстом)
   и дописывать выжимку асинхронно, чтобы задержка уведомления вообще не зависела
   от суммаризатора? Плюсы/минусы, риски «двойной карточки».

## Формат ответа
- Краткий вывод: главная причина медленности + 1 рекомендованный путь.
- Таблица локальных моделей-кандидатов с цифрами и трейдоффами.
- Конкретный план интеграции рекомендованного варианта (через localhost-сайдкар,
  как Silero): какой рантайм (llama.cpp/Ollama/MLX), какая модель+квантизация,
  ожидаемые RAM/скорость, как добиться надёжного JSON-выхода, фолбэк если модель
  не установлена.
- Запасной вариант (оптимизация текущего haiku-пути), если локальная модель не
  взлетит, с ожидаемым выигрышем.
Будь конкретным (имена моделей, размеры, команды запуска), опирайся на свежие
бенчмарки, не выдумывай цифры — если не уверен, помечай как оценку.
