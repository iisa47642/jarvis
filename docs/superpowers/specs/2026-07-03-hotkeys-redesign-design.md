# Редизайн горячих клавиш: рекордер, конфликты, приостановка команд

Дата: 2026-07-03. Статус: утверждено (визуально — через мокапы, технически — в диалоге).

## Проблема

1. Редактор хоткеев — конструктор из тоггл-чипов ⌘⌃⌥⇧ + отдельная капса основной
   клавиши (`hotkeyEditorField`). Выглядит плохо, неудобен. Появился как костыль:
   macOS-webview «съедал» ⌘/⌥-комбо до JS.
2. Диктовка — те же костыли: кнопки-пресеты (F8/F9/⌥Space/⌘⌥D/⌘K).
3. «Глобальный хоткей» в «Основном» — read-only, поменять нельзя (только сброс).
4. Во время записи сочетания глобальные шорткаты Jarvis продолжают работать:
   жмёшь ⌘⌥C, чтобы записать, — реально срабатывает «Продолжить сессию».
5. Конфликты между своими хоткеями не детектятся и не показываются: можно
   поставить одно сочетание на два действия, одно из них молча умрёт.

Гипотеза (проверяется спайком в начале реализации): пункты 1 и 4 связаны —
комбо «съедались» нашими же зарегистрированными глобальными шорткатами.
После приостановки шорткатов webview должен отдавать keydown нормально.

## Решение — UX

Единый компонент «капсула-рекордер» для ВСЕХ хоткеев (утверждён вариант «A —
инлайн-рекордер», как в Raycast/VS Code):

- Обычное состояние: капсула с клавишами (`⌘` `⌥` `C`); кнопка сброса ↺
  появляется при наведении на строку.
- Клик → режим записи: капсула пульсирует синим, текст «Нажмите сочетание…»,
  Esc — отмена, клик мимо — отмена. На время записи все глобальные хоткеи
  Jarvis приостановлены.
- Нажал комбо → применяется сразу (если валидно и нет конфликта).
- Валидация как сейчас: голая буква/цифра без модификатора запрещена,
  F-клавиши можно без модификатора.

Конфликт со своим хоткеем — «перехват только с подтверждением»:

- Новое сочетание НЕ применяется. Строка подсвечивается красным, под ней:
  «⚠ ⌘⌥J занято "Тихий режим" · Всё равно назначить».
- «Всё равно назначить» → хоткей переезжает; у старого действия — состояние
  «не назначен» (пунктирная капсула, клик по ней — запись нового).
- Esc / клик мимо → откат к прежнему сочетанию, ничего не меняется.

Вкладка «Горячие клавиши» — один общий список всех хоткеев по группам
(утверждено мокапом `keys-pane.html`):

- **Панель и сессии**: Открыть панель, Продолжить сессию, Повторить,
  Варианты ответа.
- **Звук и уведомления**: Без звука, Тихий режим.
- **Голос**: Диктовка (бейдж «push-to-talk»; дублируется в «Голосовом вводе» —
  тот же компонент).

Частные случаи:

- **Варианты ответа** (шаблон `…+{n}`): капсула показывает `⌘` `⌥` `1…9` с
  фиксированной частью «1…9». Запись: жмёшь любое комбо с цифрой — цифра
  заменяется на `{n}`, модификаторы берутся из нажатия. Комбо без цифры в
  записи этого поля невалидно.
- **«Глобальный хоткей» в «Основном»** становится редактируемым — та же
  капсула (одно действие, два места в UI).
- Кнопки-пресеты диктовки удаляются полностью.

## Решение — техника

### 1. Приостановка хоткеев: `hotkeys_suspend(on: bool)`

Новая IPC-команда. `on=true`: снять все глобальные шорткаты Jarvis — главный,
quiet, continue, repeat, mute, диктовку и набор 1…9 (если сейчас
зарегистрирован — вопрос активен). `on=false`: зарегистрировать обратно
(select-набор — только если вопрос всё ещё активен).

Страховки от «умершего» UI (иначе Jarvis навсегда без хоткеев):

- авто-возобновление по таймауту ~15 с на стороне бэкенда;
- возобновление при скрытии/закрытии панели;
- идемпотентность (повторный suspend/resume — no-op).

UI зовёт suspend при входе в запись, resume — при выходе (любым путём).

### 2. Запись комбо: спайк → основной путь → запасной

- **Спайк (первый шаг реализации)**: приостановить шорткаты и проверить,
  отдаёт ли webview ⌘/⌥-keydown в JS.
- **Основной путь** (если да): текущий `eventToAccel` + keydown-захват, как
  уже сделано у диктовки.
- **Запасной путь** (если webview всё же ест комбо): нативный местный монитор
  клавиатуры macOS (`NSEvent.addLocalMonitorForEventsMatchingMask`, через
  objc2), активен только в режиме записи, шлёт нажатия в UI Tauri-событием.

В обоих случаях чипы-тогглы и пресеты удаляются.

**Итог спайка (04.07.2026, живая проверка в дев-сборке):** при приостановленных
шорткатах webview отдаёт keydown нормально — записи проходили циклами
suspend → применение → resume без авто-ресюма по таймауту. Запасной путь
(NSEvent-монитор) не понадобился и не реализован.

### 3. Назначение и конфликты: `hotkey_assign(action, accel, steal)`

Одна атомарная команда вместо разрозненной логики в `settings_set` /
`stt_set_hotkey`:

- `action`: `panel | continue | repeat | mute | quiet | select | dictation`.
- Валидация: парсинг акселератора (для `select` — шаблон с `{n}`,
  нормализация как `normalize_select_template`).
- Конфликт со своим хоткеем (сравнение распарсенных `Shortcut`, не строк;
  для `select` — все 9 экземпляров) → ответ
  `{ ok:false, conflict: { action, label } }`, ничего не меняется.
- `steal=true` → у конфликтующего действия хоткей снимается (unregister +
  настройка `""` = «не назначен»), новое — регистрируется и сохраняется.
- «Занято системой» (регистрация не удалась) → `{ ok:false, error }`, откат
  на прежнее, как сейчас.
- Дефолты по действию известны бэкенду → сброс = `hotkey_assign(action,
  default)`, отдельной команды не нужно.

Существующие пути (`settings_set` c `hotkey`/`quietHotkey`/…,
`stt_set_hotkey`) остаются для обратной совместимости, но UI переходит на
`hotkey_assign`.

### 4. Состояние «не назначен»

Настройка = пустая строка. `register_*` при пустом акселераторе — пропуск,
`is_*_hotkey` — false. UI: пунктирная капсула «не назначен», клик — запись.

## Затрагиваемые файлы

- `src-tauri/src/ipc.rs` — `hotkeys_suspend`, `hotkey_assign`, реестр
  действий (accessor/label/default), поддержка пустых акселераторов.
- `src-tauri/src/main.rs` — регистрация команд; ресюм при скрытии панели.
- `ui/settings2.js` — единый `hotkeyRecorderField`, перестройка вкладки
  «Горячие клавиши» (группы), «Основное» и «Голосовой ввод» переводятся на
  него; удаление `hotkeyEditorField`, пресетов и старого `hotkeyField`.
- `ui/index.html` — CSS капсулы/записи/конфликта/«не назначен».
- (запасной путь) `src-tauri/src/…` — NSEvent-монитор за фичегейтом macOS.

## Тесты

- cargo: конфликт-детект (включая select-шаблон ×9 и диктовку), steal-путь,
  нормализация `{n}`, пустой акселератор в register/is_*.
- Живая проверка в дев-сборке (`npm start`): запись ⌘-комбо, конфликт+перехват,
  Esc-откат, авто-ресюм хоткеев, диктовка после смены хоткея.

## Вне скоупа

- Хоткеи вне Jarvis (детект конфликтов с другими приложениями — только
  «занято системой» от ОС, как сейчас).
- Переезд настроек/миграции — формат хранения не меняется.
