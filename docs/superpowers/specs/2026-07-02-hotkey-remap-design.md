# Переопределение всех хоткеев Jarvis (конструктор ⌘-комбинаций)

Дата: 2026-07-02 · Статус: одобрено (вариант «конструктор модификаторов»)

## Проблема

Бэкенд уже умеет перепривязывать все 5 глобальных хоткеев через `settings_set`
(`ipc.rs:299-334`: валидация «занято системой», откат, персист), но UI вкладки
«Горячие клавиши» показывает их read-only (капсулы + сброс). Запись сочетания
есть только у хоткея диктовки, причём macOS-webview «съедает» ⌘/⌥-комбо до JS —
записать их нажатием нельзя (комментарий `settings2.js:440`), для диктовки это
обходили пресет-чипами. Хоткеи выбора варианта ответа ⌘⌥1-9 захардкожены;
дефолт `selectHotkeyTemplate` существует с #38, но нигде не читается (мёртвый).

## Решение

### UX: конструктор модификаторов (без нативного захвата и новых разрешений ОС)

Для каждого хоткея — редактор: тоггл-чипы `⌘ ⌥ ⌃ ⇧` + капсула основной клавиши
(клик → «нажмите клавишу…», ловим ОДИНОЧНОЕ нажатие буквы/цифры/F-клавиши —
одиночные клавиши webview отдаёт нормально). Итоговый аксельератор собирается
из чипов + клавиши. Валидация: без модификатора допустимы только F-клавиши.
Применение сразу через `setSettings({key: accel})`; ошибку бэкенда («занято»)
показываем в капсуле и откатываем отображение. Кнопка сброса остаётся.

### Скоуп хоткеев

| Ключ | Действие | Дефолт |
|---|---|---|
| `hotkey` | Открыть панель | ⌘J |
| `continueHotkey` | Продолжить сессию | ⌘⌥C |
| `repeatHotkey` | Повторить | ⌘⌥R |
| `muteHotkey` | Без звука | ⌘⌥M |
| `quietHotkey` | Тихий режим | ⌘⌥J |
| `selectHotkeyTemplate` | Варианты ответа (цифра 1-9 фиксирована) | ⌘⌥{n} |

Хоткей диктовки не трогаем — у него уже есть рабочий редактор с пресетами
(из #18); унификация — отдельная итерация (YAGNI).

### Бэкенд: проводка selectHotkeyTemplate

- `select_template(d)` — читает `selectHotkeyTemplate`, дефолт
  `Command+Alt+{n}`; шаблон без `{n}` считается битым → дефолт.
- `select_accel(tpl, n)` — подстановка номера.
- `set_select_hotkeys` и `is_select_hotkey` работают по шаблону
  (`is_select_hotkey` получает `&Arc<Daemon>`; call-site `main.rs:99`).
- `settings_set`: при смене `selectHotkeyTemplate` — валидация (`{n}` внутри,
  экземпляр с n=1 парсится как `Shortcut`); если ⌘⌥-цифры сейчас
  зарегистрированы (вопрос активен) — снять по старому шаблону и поставить по
  новому, при провале — откат и ошибка.

### UI

- Новый `hotkeyEditorField(accel, {onApply, onReset})` в `settings2.js`
  (переиспользует `eventToAccel`, `hotkeyKeys`).
- `renderKeys`: 5 строк переводятся на редактор; добавляется строка «Варианты
  ответа» — только чипы модификаторов, капсула клавиши показывает `1…9` и не
  редактируется; сохраняет `selectHotkeyTemplate`.
- `HK_DEFAULTS` дополняется `selectHotkeyTemplate`.

## Ошибки и краевые случаи

- Конфликт двух действий на одном сочетании: `gs.register` падает (уже
  зарегистрировано нами или системой) → ошибка «Сочетание занято», откат.
- Пустой/битый шаблон в настройках → дефолт ⌘⌥{n} (мягкая деградация).
- Захват основной клавиши: Esc — отмена, клик мимо — отмена (как у диктовки).

## Тесты

- Rust: select_template (дефолт/битый шаблон), select_accel, is_select_hotkey
  по кастомному шаблону, невалидный шаблон в settings_set → err.
- UI-харнеса нет — проверка вживую после пересборки (`npm start`).
