# Jarvis · Инкремент 9: Сервис распознавания речи (STT), движок на выбор — Qwen3-ASR / Whisper

> Описательная спека без кода — реализацию агент дополнит по фактическому коду. Сделать STT не фичей одной кнопки, а переиспользуемым внутренним сервисом с собственным API, сменными движками и несколькими потребителями.
> **Скоуп v1 (по указанию пользователя 2026-06-19): только push-to-talk диктовка.** Голосовой агент-вход и A/B всех трёх движков — закладываются архитектурно, но v1-цель — диктовка по клавише как доказательство сервиса. Главное — правильный отдельный модуль, к которому обращаются и приложение, и плагины.

---

## 1. Назначение и рамки

Под голосовую фазу Jarvis нужен локальный речевой ввод. Но он нужен не одному месту, а нескольким сразу: голосовому агенту (вход «поговорить с Jarvis») и диктовке по горячей клавише (как Wispr Flow: зажал клавишу, наговорил, отпустил — текст появился в активном поле). Поэтому STT строится как **отдельный сервис внутри Jarvis с собственным API**, к которому обращаются разные части приложения, а в будущем — плагины.

Дополнительное требование — **сравнить и оставить выбор**: движок распознавания сменный, с тремя реализациями для A/B (Qwen3-ASR 0.6B по умолчанию, Qwen3-ASR 1.7B как мощнее, Whisper large-v3-turbo для сравнения), переключается настройкой в Jarvis.

Что инкремент **делает**: STT-сервис с API; сменные движки за единым интерфейсом; общий аудио-фронтенд (захват микрофона, VAD, ресемпл); потребителя «диктовка по клавише» как конкретный рабочий результат; проекцию STT в капабилити, чтобы плагины могли пользоваться через гейт; интеграцию того же API в голосового агента как второго потребителя.

Что **не делает**: не реализует wake-word («Jarvis» — отдельный слой-триггер агента), не строит полный голосовой цикл с TTS-ответом (это связка с инкрементами 7 и 8), не делает живую потоковую транскрипцию по словам (v1 — батч; стриминг отложен), не уходит в облако (локально — это весь смысл).

**Локально и приватно — жёстко.** Аудио не покидает машину ни при каком движке. Сеть для распознавания не используется.

---

## 2. Главная архитектурная идея — зафиксирована

1. **STT — переиспользуемый сервис с собственным API, а не код внутри одного потребителя.** Диктовка, голосовой агент и будущие плагины обращаются к одному сервису. Сервис — источник истины по распознаванию.
2. **Движок сменный за единым интерфейсом.** Три реализации для A/B; пользователь флипает настройку `engine` и сравнивает на своей речи, не трогая потребителей. Список движков расширяем.
3. **Сервис проецируется в капабилити** (из инкремента 8), поэтому плагины пользуются STT через тот же гейт/гранты. Внутренние потребители (диктовка, агент) зовут сервис напрямую через внутренний биндинг.
4. **Аудио-фронтенд общий** (захват, VAD, ресемпл); движок — сменная часть позади него. Потребители различаются триггером и приёмником результата, но делят фронтенд и сервис.
5. **Code-switching с пословной верностью скрипту — требование первого класса** (см. §11), а не побочное свойство.

## 3. Слои и потоки

```
ПОТРЕБИТЕЛИ
  Диктовка по клавише        Голосовой агент (вход)        Плагины (через гейт/грант)
        │ внутренний биндинг        │ внутренний биндинг           │ капабилити stt.*
        ▼                            ▼                              ▼
┌──────────────────────────────────────────────────────────────────────┐
│  STT-СЕРВИС — собственный API (transcribe / опции / событие готовности)│
│  ├─ Аудио-фронтенд (общий): захват микрофона · VAD · ресемпл 16к моно  │
│  └─ Движок (сменный, за трейтом)                                       │
│        Qwen3-ASR 0.6B (sidecar, MLX) ← по умолчанию                    │
│        Qwen3-ASR 1.7B (sidecar, MLX) ← мощнее                          │
│        Whisper large-v3-turbo (нативно, whisper-rs / Metal)           │
└──────────────────────────────────────────────────────────────────────┘
```

Поток словами: триггер потребителя открывает захват → аудио-фронтенд буферизует и режет фразу (по VAD или по отпусканию клавиши) → ресемпл под модель → активный движок транскрибирует → текст (с пословной разметкой языка, где движок даёт) возвращается потребителю. Диктовка вставляет текст в активное поле ОС; агент отдаёт текст в свой цикл; плагин получает результат через капабилити и гейт.

## 4. Зафиксированные решения — не пересматривать

- **Единый интерфейс движка** (трейт `SttEngine`) с тремя реализациями; выбор по конфигу на старте; горячую смену не делаем (требует перезапуска сервиса/демона).
- **Qwen3-ASR — Python-сайдкар** (MLX на Apple Silicon), держит модель в памяти, демон шлёт PCM, получает результат. Причина: нет нативного Rust-инференса. Размер модели (0.6B/1.7B) — параметр конфига.
- **Whisper — нативно** через whisper-rs/whisper.cpp (Metal), в процессе, без сайдкара. Причина: ниже латентность для диктовки и одиночный бинарь. Асимметрия транспортов допустима — трейт даёт единый API, пользователь сравнивает качество вывода, а не транспорт.
- **Батч — основной режим v1** (зажал/наговорил/отпустил → транскрипция). Обе целевые задачи (ход агента, push-to-talk-диктовка) батч-дружелюбны. Живой стриминг по словам — отложен (модели батчевые; форсировать стриминг не усложняем).
- **Аудио-фронтенд общий** (cpal-захват, Silero VAD для эндпойнтинга в hands-free, отпускание клавиши как эндпойнт в push-to-talk, ресемпл в 16кГц моно).
- **STT-капабилити гейтятся**: использование микрофона плагином — приватно-чувствительно, поэтому плагину нужен явный грант на STT; внутренние потребители (диктовка, агент) зовут напрямую.
- **Локально, без сети.** Любой движок — офлайн.
- **Дефолт-конфиг настроен под смешанную RU/EN речь** (см. §11).

## 5. STT-сервис и его API

**Что это.** Внутренний сервис-источник истины по распознаванию. У него собственный типизированный API для внутренних потребителей и проекция в капабилити для плагинов.

**API (в терминах контракта, не кода):**
- **transcribe(аудио, опции) → результат** — разовая транскрипция готового буфера. Результат: текст; где движок даёт — пословная/посегментная разметка языка и уверенность.
- **опции** — доминирующий язык (пин), режим задачи (транскрипция, не перевод), подсказки; каждый движок мапит их по-своему (для Whisper это критично, см. §11).
- **сессия захвата** — открыть/закрыть поток с микрофона для потребителя (push-to-talk держит её на время удержания клавиши; hands-free закрывает по VAD-эндпойнту).
- **событие готовности/ошибки** — потребитель узнаёт, что текст готов или что движок недоступен.

**Проекция в капабилити** (инкремент 8): например `stt.transcribe` и управление сессией захвата — как капабилити класса, требующего гранта на микрофон для плагинов. Внутренние потребители используют сервис через внутренний биндинг (через гейт ради единого аудита, но это read-ish, низкий риск).

---

## 6. Движки (трейт + три реализации)

- **Трейт `SttEngine`**: контракт `transcribe(аудио, опции) → результат`. Опции включают доминирующий язык, режим задачи, подсказки. Выбор реализации — по конфигу.
- **Qwen3-ASR 0.6B (по умолчанию) и 1.7B** — Python-сайдкар на MLX (Apple Silicon). Сильнейшая мультиязычность и устойчивость к смешанной/акцентированной речи; пословную верность скрипту даёт наиболее естественно (LLM-декодер «знает» написание). 1.7B — когда нужно качество выше ценой ресурса. Цена: сайдкар, дозревающая экосистема.
- **Whisper large-v3-turbo** — нативно через whisper-rs/whisper.cpp (Metal). Проверенный, нативная пунктуация, огромная экосистема, Rust-нативный (без сайдкара, ниже латентность). **Требует правильной конфигурации под смешанную речь** (см. §11), иначе сваливается в перевод всего в один язык.
- Список движков расширяем; Parakeet сознательно не включаем — его per-utterance-детект ломает пословную верность скрипту (см. §11).

---

## 7. Аудио-фронтенд (общий)

- Захват микрофона (cpal), выбор входного устройства из конфига.
- VAD (Silero) для эндпойнтинга в hands-free-режиме; в push-to-talk эндпойнт — отпускание клавиши.
- Буферизация и ресемпл в формат модели (16кГц моно).
- Один фронтенд на оба потребителя; движок позади него.

---

## 8. Потребитель 1: диктовка по клавише (конкретный результат v1)

- Глобальная горячая клавиша (удержание или тоггл — из конфига).
- Пока клавиша активна — захват микрофона; на отпускании — транскрипция активным движком.
- Результат **вставляется в активное поле ОС** (системно, в любое приложение под курсором). Механизм вставки — забота потребителя (Accessibility/синтетическая вставка/буфер обмена), как обсуждалось для Wispr-сценария; здесь не углубляемся, но диктовка обязана текст вставлять.
- Это эталон, доказывающий сервис: проходит весь путь захват → движок → текст наружу.

---

## 9. Потребитель 2: голосовой агент (вход)

- Тот же API сервиса: триггер (хоткей агента; wake-word — отдельный слой, отложен) открывает захват → транскрипция → текст **отдаётся в цикл агента** (агент из инкремента 8), а не вставляется в приложение.
- В v1 API готов и потребитель определён; полный голосовой цикл (wake-word, TTS-ответ из инкремента 7) — собирается, когда строится голосовой агент. Здесь — входная половина.

---

## 10. Плагины и гейт

- STT доступен плагинам как капабилити через гейт/гранты (инкремент 8). Использование микрофона — приватно-чувствительно, поэтому плагину требуется **явный грант на STT**; без гранта капабилити не видна и недоступна.
- Это делает STT платформенным сервисом: будущий плагин (например, голосовая команда под свой сценарий) пользуется тем же распознаванием, не получая прямого доступа к микрофону в обход гейта.

---

## 11. Code-switching — требование первого класса

Целевой кейс: русская речь с английскими тех-вкраплениями в одной фразе, с **пословной верностью скрипту** — русское кириллицей, английское латиницей, без схлопывания всей фразы в один язык.

- **Дефолт-конфиг настроен под это:** режим задачи — транскрипция (никогда не перевод); доминирующий язык пиннится на русский (доминанта речи — русский, английское вкраплено); для Whisper — большая модель (turbo), иначе смешанная транскрипция деградирует.
- **Whisper-движок обязан** прокидывать эти опции (transcribe + пин ru), чтобы не уплывать в сплошной английский — известная ловушка Whisper на смешанном аудио.
- **Qwen3-движок** держит пословную верность естественнее (LLM-декодер), но дефолт-опции применяются и к нему.
- Движки с per-utterance-детектом языка (Parakeet) **не подходят** — они выбирают один язык на кусок и калечат смешение; поэтому исключены.

---

## 12. Конфиг и панель

- Конфиг `~/.jarvis/`: выбор движка (`qwen3-0.6b` | `qwen3-1.7b` | `whisper-turbo`); per-engine (путь/размер модели, доминирующий язык, режим задачи); общий аудио-фронтенд (входное устройство, VAD, sample rate); настройки потребителей (хоткей диктовки, триггер агента). Переключение движка — рестарт сервиса.
- Панель: выбор активного движка, тест распознавания (наговорить и увидеть результат текущим движком), индикатор статуса (движок доступен / сайдкар поднят), разрешения (микрофон; для диктовки — Accessibility на вставку).

---

## 13. Установщик

- Qwen3-сайдкар: venv + MLX + загрузка моделей (0.6B и 1.7B) в кэш `~/.jarvis/`. Явно залогировать вес установки.
- Whisper: загрузка модели large-v3-turbo (whisper.cpp), нативно.
- Ставятся все три, чтобы A/B был возможен; активный выбирается конфигом. Отсутствует зависимость одного движка → не падать, пометить в `status`, остальные доступны. Идемпотентно/бэкап/atomic как в остальных частях.
- `status` по каждому движку: установлен, доступен (сайдкар отвечает / модель на месте), какой активен; плюс разрешения микрофона/Accessibility.

---

## 14. Что переиспользуется / не переписываем

- Паттерн сайдкара — тот же, что у Silero-TTS (инкремент 7); аудио-проигрывание/захват на Rust — те же подходы (cpal). Гейт/гранты/капабилити — из инкремента 8, STT встраивается как ещё один сервис-капабилити. Голосовой агент — потребитель из инкремента 8, ему добавляется голосовой вход. Новый код — STT-сервис, движки, аудио-фронтенд, потребитель-диктовка; всё поверх существующего.
- При расхождении фактической формы (API сайдкара Qwen3 MLX, опции whisper-rs, форматы моделей) со спекой — остановиться и описать, не выдумывать обход молча.

## 15. Объём v1 и дальше

**В v1:** STT-сервис с API; трейт движка + три реализации (Qwen3 0.6B/1.7B сайдкар, Whisper turbo нативно); общий аудио-фронтенд (захват, VAD, ресемпл); потребитель-диктовка по клавише со вставкой текста; проекция в капабилити с грантом на микрофон для плагинов; настройка выбора движка и тест в панели; дефолт под смешанную RU/EN речь.

**Дальше (заложить, не строить):** wake-word как триггер агента; полный голосовой цикл (STT-вход + агент + TTS-ответ); живая потоковая транскрипция по словам; голосовой потребитель агента целиком; дополнительные движки.

## 16. Приёмочные сценарии

1. Зажал хоткей, наговорил по-русски, отпустил → кириллический текст вставлен в активное поле под курсором.
2. **Ключевой тест code-switching:** наговорил «давай запушим в main и проверим CI перед мержем» → «main», «CI», «push» остаются латиницей внутри русской фразы, вся фраза **не флипается в английский**. Проверяется на каждом движке.
3. Сменил в конфиге движок (`qwen3-0.6b` → `whisper-turbo` → `qwen3-1.7b`), рестарт → тот же API, другой движок, пользователь сравнивает качество на одной речи **без правок потребителей**.
4. Голосовой агент использует тот же API: захват → транскрипция → текст уходит в цикл агента (а не вставляется в приложение).
5. Плагин с грантом на STT вызывает `stt.transcribe` через гейт → получает транскрипцию; плагин без гранта капабилити не видит.
6. Qwen3-сайдкар недоступен при активном Qwen3 → диктовка падает мягко, в логах причина; переключение на нативный Whisper восстанавливает работу; демон жив.
7. Офлайн-проверка: при выключенной сети распознавание работает на любом движке (локальность подтверждена).
8. Whisper-движок с дефолт-опциями (transcribe + пин ru) держит английские термины латиницей, не уплывая в сплошной английский.
9. Латентность push-to-talk: от отпускания клавиши до появления текста — в приемлемых рамках; замерить на каждом движке (ожидаемо Whisper-нативный ниже по латентности, Qwen3 выше по качеству mixed).

---

## 17. Не делать

- Wake-word («Jarvis») — отдельный слой-триггер агента, не здесь.
- Полный голосовой цикл с TTS-ответом — связка инкрементов 7+8+9, не этот инкремент.
- Живой стриминг по словам — батч в v1.
- Облачный STT-фоллбек — локально по дизайну.
- Движки с per-utterance-детектом языка (Parakeet) — ломают пословную верность скрипту.
- Глубокий разбор механизма вставки текста — диктовка его использует, но это известная забота потребителя.

---

## 18. Качество

- Раздельные модули: STT-сервис, аудио-фронтенд, движки (за трейтом), потребители; смена движка не трогает потребителей.
- Тесты без живого микрофона: маппинг опций (transcribe/пин языка) в каждом движке, выбор движка по конфигу, генерация результата с пословной разметкой языка, проекция в капабилити с проверкой гранта.
- Fail-safe на сбой/отсутствие движка (особенно сайдкара): таймауты, изоляция, демон не падает, Claude Code не затронут.
- Локальность проверяема (работа офлайн).
- README: архитектура (STT = переиспользуемый сервис с API, движок сменный, потребители на одном API), как переключать движок и сравнивать, **явно** — размен Qwen3 (лучший mixed/скрипт, сайдкар, тяжелее) vs Whisper-turbo (Rust-нативный, ниже латентность, требует пина ru под смешанную речь); и что wake-word и полный голосовой цикл — следующие фазы.
- При расхождении фактического кода/форм со спекой — остановиться и описать, не переписывать молча.
