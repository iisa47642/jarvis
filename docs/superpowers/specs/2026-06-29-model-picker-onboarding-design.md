# Дизайн: выбор и мультизагрузка моделей с прогрессом

Дата: 2026-06-29 · Ветка: `feat/model-picker-onboarding`

## Проблема

На первом запуске (онбординг) нельзя выбрать, какие модели скачать. Онбординг
ставит только хуки/транспорт/Silero; веса STT (Whisper, Qwen) и wake-word качаются
потом поштучно из панели «Модели». Пользователь хочет: при установке выбрать
нужные модели (Whisper, Qwen, голосовая активация, голос) и скачать всё сразу,
видя прогресс. Дополнительно — то же «скачать выбранное» в панели «Модели».

Сопутствующий факт: прогресс-бар в панели есть, но на практике не появлялся —
загрузка Whisper падала на коннекте из-за proxy-бага (прокси читался из
верхнеуровневого `proxy`, а сохранялся в `service.proxy`). Баг исправлен отдельно
(PR #31, `Settings::proxy()`); эта фича опирается на исправленный прокси.

## Решение (подход A: единый оркестратор + единые события)

### Backend (Rust)

**Каталог моделей** — единый список устанавливаемых моделей. Для каждой:
`id`, `kind` (`stt` | `wake` | `voice` | `runtime`), человекочитаемый `label`,
примерный размер, и какой существующий загрузчик её ставит:

| id            | kind    | загрузчик                              | размер   |
|---------------|---------|----------------------------------------|----------|
| `whisper-turbo` | stt   | `install_whisper`                      | ~574 МБ  |
| `qwen3-0.6b`  | stt     | `install_stt_sidecar` + `preload_qwen` | ~1 ГБ(+venv) |
| `qwen3-1.7b`  | stt     | `install_stt_sidecar` + `preload_qwen` | ~1 ГБ(+venv) |
| `hey_jarvis`  | wake    | `install_wakeword`                     | ~3.5 МБ  |
| `silero`      | voice   | `install_silero`                       | ~1 ГБ    |

**Планировщик (чистая функция, тестируемая):** `plan_install(ids) -> Vec<Task>`.
- Раскрывает зависимости: выбор `qwen3-*` добавляет рантайм-venv первым шагом
  (если ещё не установлен).
- Канонический порядок: рантаймы/venv → веса → мелочь.
- Дедуп: уже установленное не включается.

**Команда `models_install(app, ids: Vec<String>)`:** один фоновый поток.
- `let proxy = d.settings.proxy();` (исправленный читатель).
- Идёт по `plan_install(ids)`; для каждой задачи зовёт её загрузчик с
  progress-замыканием, которое эмитит `model_install_progress { id, pct, state, msg }`.
- После задачи — `model_install_done { id, ok, error }`.
- В конце — `models_install_all_done { results }`.
- Сбой одной задачи **не прерывает** очередь (остальные качаются дальше).

Существующие per-model команды (`stt_install_whisper`, `stt_install_qwen`, …)
остаются в коде, но UI больше их не вызывает напрямую — всё идёт через
`models_install([id])`, чтобы прогресс был единым.

### Frontend

**Онбординг — шаг «Модели».** После ядра (хуки/транспорт) — чеклист с размерами:
- ☑ Whisper large-v3-turbo (~574 МБ)
- ☐ Qwen3-ASR + радио 0.6B / 1.7B (~1 ГБ)
- ☑ Голосовая активация / wake-word (~3.5 МБ)
- ☑ Голос Silero (~1 ГБ)

(Дефолтные галочки — стартовая раскладка, не догма.) Кнопка «Скачать выбранное»
→ `models_install(ids)`. Качается **в фоне**: окно закрываемо, у каждой выбранной
модели строка с `progressBar`; статус также виден в панели «Модели».

**Панель «Модели».** К каждой не-скачанной строке — чекбокс; над группой — кнопка
«Скачать выбранное (~N ГБ)» → та же `models_install`. Плейсхолдер прогресса в
строке (`[data-model=id]`) подключается к единым событиям по `id`.

### Поток данных

```
JS models_install(ids)
  → Rust: поток по plan_install(ids)
    → загрузчик(model) — progress → emit model_install_progress{id,pct}
       → JS рисует progressBar в строке [data-model=id]
    → emit model_install_done{id,ok,error} → строка done/ошибка(+«Повторить»)
  → emit models_install_all_done → перерисовка панели / кнопка «Готово»
```

### Граничные случаи

- **Сбой сети/прокси** у одной модели — `done{ok:false}` → строка с ошибкой и
  «Повторить»; очередь не прерывается.
- **Закрытие окна** в процессе — фоновый поток живёт дальше, статус в панели.
- **Уже установлено** — пропуск (загрузчики идемпотентны + дедуп в плане).
- **Qwen без рантайма** — план сам подставляет `install_stt_sidecar` первым.

### Тестирование

- Юнит-тесты на чистый `plan_install`: раскрытие зависимости qwen→runtime,
  канонический порядок, дедуп уже установленного, маппинг id→kind/размер.
- Реальные загрузки/сеть не тестируем.
- Существующие тесты остаются зелёными.

## Вне объёма (YAGNI)

- Параллельные загрузки (последовательность бережёт сеть и проще по прогрессу).
- Докачка с резюмом после обрыва (есть «Повторить» — перекачивает с нуля).
- Выбор Intel-сборок/иных STT-движков.
