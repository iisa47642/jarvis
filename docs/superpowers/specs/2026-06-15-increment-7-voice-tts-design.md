# Jarvis · Инкремент 7: Голос (TTS, движок на выбор — Silero / Piper)

> Дизайн-документ. Статус: утверждён к реализации. Дата: 2026-06-15.
> Реализация фазируется: **Фаза 1 — ядро + Piper** (этот план), **Фаза 2 — Silero-сайдкар + Python-установщик** (отдельный план поверх готового трейта).

## Контекст и граница

Jarvis — монитор сессий Claude Code (демон на Rust + Tauri 2). Инкремент 7 добавляет **голосовой вывод**: после структурного события демон коротко проговаривает по-русски, что произошло, локальным TTS. Движок выбирается конфигом; обе реализации живут за единым трейтом, чтобы A/B был честным, а переключение — флагом, не переписыванием.

**ГЛАВНАЯ ГРАНИЦА (зафиксирована):** это **вывод-only** — «Jarvis говорит». Микрофон, wake-word, STT, VAD, эхоподавление, half-duplex, голосовой ответ агенту — **следующая фаза, out of scope**. Ввод не добавляется ни в каком виде.

Композитор фразы — **шаблонный** (структурные сигналы → текст), не LLM. LLM-суммаризатор подключается позже как альтернативная реализация того же трейта; **здесь размечается шов, сама модель не делается**.

## Расхождения со спекой (зафиксированы при ресёрче на живом коде)

1. **Конфиг — JSON, не TOML.** Проект хранит всё в `~/.jarvis/settings.json` (конвенция `plugins.*`). Voice-конфиг — новый блок `"voice"` в этом же файле; новый формат/парсер не вводим. *(Решение пользователя.)*
2. **`last_assistant_message` берётся из транскрипта, не из payload `Stop`.** В Rust-порте финальный ответ уже читается через `transcript::full_final_reply` (используется в `done_summary`). Композиторский сигнал «текст ответа» = transcript-read.
3. **Долгоживущего supervised-процесса в проекте ещё нет.** Демон спавнит только короткие команды (tmux, claude). Piper (subprocess на реплику) ложится на это сразу; Silero-сайдкар (живой процесс, health-check, restart) — новая инфраструктура, отнесена в Фазу 2.

Если поведение Piper/Silero на живой системе разойдётся с этим документом (флаги `piper`, формат модели, payload сайдкара) — остановиться и описать расхождение, не выдумывать обходной путь молча.

## Архитектура: модуль `src-tauri/src/voice/`

Общий слой не зависит от движка. Различается только реализация трейта.

- **`engine.rs`** — `trait TtsEngine`:
  ```rust
  pub trait TtsEngine: Send + Sync {
      /// текст → WAV-байты (или PCM). Ошибка — fail-safe, демон не падает.
      fn synthesize(&self, text: &str, voice: &VoiceSel) -> Result<Vec<u8>, TtsError>;
      /// прогрев на старте демона (первый прогон/загрузка), чтобы первая
      /// реальная реплика не ловила холодный старт.
      fn warmup(&self) {}
      /// доступность для `status` (бинарь/модель на месте, сайдкар отвечает).
      fn available(&self) -> bool;
      fn name(&self) -> &'static str;
  }
  ```
  Реализации: **`PiperEngine`** (Фаза 1), **`SileroEngine`** (Фаза 2). Выбор — по конфигу на старте; горячую смену не делаем (смена `engine` требует перезапуска демона).

- **`composer.rs`** — `trait Composer { fn compose(&self, sig: &SpeechSignals) -> Option<Utterance>; }` + **`TemplateComposer`** (чистая функция). `Utterance { text: String, priority: Priority, dedup_key: String, coalesce_group: Option<String> }`. **Шов под LLM:** альтернативная реализация трейта (читает `last_assistant_message`) подставляется без изменения вызывающего кода; здесь не пишется.

- **`numerals.rs`** — русское согласование числительных и разворот единиц в слова: `одна задача` / `две задачи` / `пять задач`; `изменён один файл` / `изменено три файла`; `через два часа четырнадцать минут`. Чистые функции, покрыты тестами.

- **`queue.rs`** — сериализованная очередь речи: одна реплика за раз, приоритет, дедуп, коалесцирование бэклога, прерывание текущей.

- **`player.rs`** — проигрывание на Rust (`rodio`, default-output). Прерываемое: высокоприоритетная реплика останавливает текущий sink.

- **`config.rs`** — чтение/дефолты `voice`-блока из `settings.json`.

- **`mod.rs`** — сервис `Voice` (живёт в `Daemon`): держит engine + composer + queue + player + config; публичный метод `speak(signals: SpeechSignals)`. Всё внутри fail-safe (любая ошибка → лог + тишина, демон и Claude Code не затронуты).

## Поток данных

```
daemon хендлер события (stop / notification / stop-failure)
  └─ собрать SpeechSignals из УЖЕ имеющегося состояния:
       project, model, board(инкр.6), last_cmd+файл, diff-статы,
       transcript::full_final_reply, текст Notification, reset(инкр.5)
        └─ Voice::speak(signals)
             └─ Composer::compose → Option<Utterance>(text, priority, dedup, coalesce)
                  └─ Queue (сериализация, приоритет, дедуп, коалесцирование)
                       └─ TtsEngine::synthesize(text) → WAV
                            └─ Player (default-output, прерываемый)
```

Новых хуков/событий не добавляем — потребляем уже подключённые (`pre-tool`/`post-tool`/`prompt`/`stop`/`stop-failure`/`notification`/`session-end`).

`SpeechSignals` — структура с `Option`-полями (defensive): отсутствие сигнала не ломает композицию, просто понижает её до следующего уровня.

## Композитор: логика и согласование

Приоритет сигналов: **task-доска > diff-статы > голый факт события**.

- **`Stop`** → есть доска: «{project}: {N словами} из {M} задач, сейчас {активная}»; иначе diff: «{project} готов, изменено {N словами} файл(а/ов)»; иначе: «{project} закончил».
- **`Notification`** → «{project} ждёт — {суть}» (всегда, **высокий приоритет**).
- **`StopFailure`** → «{project} упёрся в лимит, сброс через {время словами}».

Реплика — **1–2 предложения**, длинный `last_assistant_message` обрезается композитором (не зачитывается целиком). Числа/единицы разворачиваются в слова внутри `numerals.rs` — не полагаемся на TTS.

## Очередь речи

Сериализованная, одна реплика за раз (никакого хора).
- **Приоритет:** `Notification`/`StopFailure` (нужен человек) **выше** `Stop` (готово).
- **Дедуп:** повторный idle-`Notification` не озвучивается дважды (по `dedup_key`).
- **Коалесцирование:** при заторе `Stop`-реплики сливаются в одну («Пиксела и Рекрю закончили»), а не зачитываются пачкой по одной (по `coalesce_group`).
- **Прерывание:** приход высокоприоритетной реплики прерывает текущую низкоприоритетную.

## Движок Piper (Фаза 1)

Subprocess бинаря `piper`: текст в stdin → WAV в stdout, модель `.onnx` + `.json` из конфига. Прогрев на старте демона. Fail-safe: нет бинаря/модели/ошибка запуска → `TtsError`, демон жив, причина в лог. Нативный путь через `ort` (ONNX in-process) — возможная будущая оптимизация, **не первый заход**.

## Движок Silero (Фаза 2 — здесь только шов)

В Фазе 1: трейт готов, `SileroEngine` — заглушка, которая при `engine="silero"` возвращает `TtsError::NotInstalled` (демон молчит, внятная причина в лог), `available() == false`. Полная реализация (Python-сайдкар: FastAPI/сокет, модель `v5_ru` в памяти, venv + torch, managed-процесс, health-check, restart) + установщик — **отдельный план Фазы 2**.

## Проигрывание

`rodio` на системный output по умолчанию. Декод WAV — встроенный в rodio (при нужде — `hound`). Прерывание текущей реплики: остановка активного `Sink` при поступлении высокоприоритетной.

## Конфиг (`voice`-блок в `settings.json`)

```json
"voice": {
  "engine": "piper",            // "piper" | "silero"
  "speaker": "ru_RU-...",       // спикер Silero (aidar/baya/…) или имя голоса Piper
  "voicePath": "~/.jarvis/voices/ru.onnx",
  "sampleRate": 24000,
  "events": { "notification": true, "stopFailure": true, "stop": true, "subagentStop": false, "sessionEnd": false },
  "verbosity": "short",         // "short" («готов») | "descriptive»
  "mute": false
}
```
Дефолты по событиям: `Notification`/`StopFailure`/`Stop` — вкл; `SubagentStop`/`SessionEnd` — выкл. Битый блок → дефолты, молча (как остальной settings.rs).

## Меню-бар (`tray.rs`)

- `CheckMenuItem` **«Без звука»** — мгновенно глушит очередь речи (флаг `mute`); баннеры-уведомления при этом остаются.
- Пункт **«Тест голоса»** — проговаривает образец текущим движком/спикером.

## Установщик (`setup.rs`)

Фаза 1 — только Piper: установка бинаря `piper` + одного русского голоса (`.onnx` + `.json`) в `~/.jarvis/`. Идемпотентно, atomic-write, бэкап — как остальные части установщика. Отсутствует зависимость → не падать, пометить в `status`.

`status` показывает по каждому движку: установлен ли, доступен ли (бинарь+модель на месте / сайдкар отвечает), какой активен.

Silero-шаги установщика (venv + torch + модель, явный лог веса PyTorch) — Фаза 2.

## Fail-safe

Сбой/отсутствие любого движка: демон не падает, Claude Code не затронут, паника невозможна, причина — в лог. Озвучка — побочный эффект, никогда не на критическом пути обработки события.

## Тестирование (без живого аудио)

- **Композитор:** выбор сигнала по событию (доска > diff > факт); русское согласование числительных и единиц времени; обрезка длинного текста до 1–2 предложений.
- **Очередь:** сериализация; приоритет (ждёт > готово); дедуп повторов; коалесцирование `Stop`-бэклога.
- **Движок/проигрывание:** мокаются (тесты не требуют звуковой карты).

## Новые зависимости

- `rodio` — проигрывание (Фаза 1).
- `hound` — опционально, ручной декод WAV.
- `reqwest`/HTTP-клиент — только Фаза 2 (клиент к Silero-сайдкару).

## Out of scope
