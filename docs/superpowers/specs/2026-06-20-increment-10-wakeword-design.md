# Инкремент 10 — Wake-word (+ шов под «только мой голос») — дизайн

> Реализация поверх инкр.9 (STT). Ветка `increment-10-wakeword` ← `increment-9-stt`.
> Стиль: описательный дизайн; код дополняет агент по фактическому коду ветки.
> Подтверждённый пользователем подход: **единая зона ответственности** (один владелец
> аудио-захвата + веер потребителей), доступ для плагина — через гейт.

## 0. Главная находка и рамка

Инкр.9 уже даёт **push-to-talk аудио-фронтенд** (`src/stt/audio.rs`):
`CaptureSession::start(device)` → копит весь PCM на отдельном потоке (cpal `!Send`) →
`finish()` → downmix→моно→ресемпл 16к. Плюс уже есть: микрофонный энтайтлмент
(`com.apple.security.device.audio-input`) и `NSMicrophoneUsageDescription` в `Info.plist`,
сервис `SttService` (трейт движка, qwen3/whisper), потребитель `Dictation`, капабилити
`stt.transcribe` (Control/Trusted, агент запрещён поимённо, плагину нужен явный грант).

Чего НЕТ и что добавляет инкр.10 (дельта):
1. **Общий always-on источник** — один владелец захвата, непрерывный, с **веерной раздачей**
   16к-моно-кадров нескольким потребителям; **кольцевой буфер-преролл**; **жёсткий mute у
   источника**; состояния `idle/listening/muted/denied/no-device` + индикатор «слушаю».
2. **Wake-word детектор** за трейтом: дефолтный инертный стаб + реальный openWakeWord
   (мел→эмбеддер→`hey_jarvis`) через `ort`, за cargo-фичей. Событие wake несёт преролл.
3. **Шов верификации** говорящего: трейт `SpeakerVerifier` + `NullVerifier` (всегда проходит,
   verification по умолчанию выкл), точка вызова строго после wake.
4. **Интеграция**: wake → STT-захват (с прероллом) → агент; капабилити статуса; конфиг;
   панель; установщик моделей; проверка разрешения микрофона.

`CaptureSession`-API сохраняется бит-в-бит, чтобы `Dictation` (PTT инкр.9) не пострадал —
внутри он становится подписчиком общего источника.

## 1. Архитектура — единая зона ответственности (Подход 1)

```
            ┌──────────────── AudioHub (единственный владелец) ───────────────┐
 микрофон → │ capture-поток (cpal Stream !Send)                                │
            │   callback: f32-конверсия → [mute? skip] → mpsc(native)          │
            │ proc-поток: downmix→моно → стрим-ресемпл 16к → кадры 80мс(1280)  │
            │   → preroll-ring (≈2с) → ВЕЕР подписчикам                        │
            │ state: idle|listening|muted|denied|no-device → emit в панель     │
            └───────────────┬───────────────────────────┬─────────────────────┘
              подписка(всегда)│                          │подписка(по требованию)
                              ▼                          ▼
                  WakeWord (стадия 1)            CaptureSession (PTT/STT)
                   feed 80мс → score             копит кадры → finish()→16к PCM
                   ≥порог (+debounce,warmup)      (Dictation инкр.9 — без изменений)
                              │ wake + преролл
                              ▼
                  SpeakerVerifier (стадия 2, ШОВ)  NullVerifier → проходит всегда
                              │ принято
                              ▼
                  Действие wake: открыть STT-захват (преролл) → transcribe → агент
```

**Почему так.** Один cpal-стрим на устройство (CoreAudio не любит два input-стрима на одном
устройстве); один ресемпл; преролл бесшовно отдаётся из общего кольца; mute — в одной точке
(доверие). Тяжёлая биометрия — только ПОСЛЕ wake, не always-on.

### Потоковая модель и realtime-safety
- **capture-поток** владеет `cpal::Stream` (он `!Send`) и не покидает поток (как в инкр.9).
  В callback — минимум: конверсия в f32, проверка `muted` (skip при mute), `push` в канал.
- **proc-поток** делает downmix + **стрим-ресемпл** (персистентный `FftFixedIn`, кормим
  чанками) + нарезку на кадры 80мс + запись в preroll-ring + рассылку подписчикам.
- **Чистое ядро `Pipeline`** (downmix→ресемпл→нарезка→preroll→fan-out) отделено от
  cpal/потоков и **тестируется на синтетике без живого микрофона**.

### Веер (fan-out)
Реестр подписчиков `Mutex<Vec<Subscriber>>`, каждый — `mpsc::Sender<Arc<[f32]>>` (кадр в
`Arc`, дешёвый клон). proc-поток рассылает каждый кадр всем; мёртвых (Receiver уронен) —
вычищает. Для 1–3 потребителей это проще и достаточнее tokio::broadcast; без новых зависимостей.

### Жёсткий mute (доверие)
`AtomicBool muted` проверяется **в callback** (кадр не попадает в канал) и защитно в proc.
Muted ⇒ ни один подписчик не получает аудио ⇒ wake-word не может сработать. Индикатор отражает
`listening && !muted`.

### Жизненный цикл (on-demand)
`AudioHub` держит счётчик «спроса» = (wake включён?) + (число активных capture-сессий).
demand 0→>0 ⇒ старт capture+proc потоков; demand >0→0 ⇒ останов (drop Stream).
Диктовка-только (wake выкл) ⇒ микрофон холодный вне PTT. Wake вкл ⇒ горячий (это присуще).

## 2. Слой Wake-word (за трейтом)

```rust
trait WakeWordEngine: Send {
    fn name(&self) -> &str;
    /// Скормить ровно один 80мс-кадр (1280 f32 @16к). Вернуть текущий скор 0..1
    /// (или None, пока окно не набралось). Stateful.
    fn push_frame(&mut self, frame: &[f32]) -> Option<f32>;
    fn reset(&mut self);
}
```

- **`StubEngine`** (дефолт, без ML): `push_frame → Some(0.0)` всегда — присутствует, но **инертен**
  (никогда не будит). Безопасно по умолчанию. Полный конвейер wake→verify→action тестируется
  через тест-only `FixtureEngine`, срабатывающий по маркеру.
- **`OwwEngine`** (фича `wakeword-ort`): openWakeWord через `ort = "=2.0.0-rc.10"` (пин ради
  совместимости с будущим `voice_activity_detector`, у которого `links=onnxruntime`). Конвейер:
  - chunk 1280 сэмплов (80мс) → мел на последних 1760 (1280+3·160) → squeeze [8,32] →
    трансформ `x/10+2` → mel-ring (cap 970);
  - окно `mel[-76:]` → `[1,76,32,1]` → embedding → squeeze [96] → emb-ring (cap 120);
  - `emb[-16:]` → `[1,16,96]` → классификатор `hey_jarvis` → скаляр 0..1.
  - Warm-up: первые 5 кадров → 0.0. Имена входов: `input` / `input_1` / читать у классификатора.
  - Модели отсутствуют ⇒ `OwwEngine::load` → Err ⇒ движок не строится ⇒ стаб ⇒ fail-safe.
  - Сессии single-thread (intra=1, inter=1), CPU EP (CoreML опционально позже).

**`WakeWord` сервис** (Arc в Daemon): держит `Box<dyn WakeWordEngine>`, конфиг (вкл, порог,
debounce, путь модели). При включении — подписывается на хаб, гоняет consumer-поток: кормит
кадры движку, скор≥порога N кадров подряд (debounce) ⇒ снимок преролла ⇒ verifier ⇒ action.
Супервизия tick’ом; dispose на выходе. Сбой движка/паника изолированы (catch, лог) — демон жив.

## 3. Слой верификации (ШОВ)

```rust
trait SpeakerVerifier: Send + Sync {
    fn enabled(&self) -> bool;
    /// Сверить аудио срабатывания (преролл+хвост) с профилем → совпадение 0..1.
    fn verify(&self, audio_16k: &[f32]) -> f32;
    /// Зачислить профиль из нескольких произнесений.
    fn enroll(&self, utterances: &[Vec<f32>]) -> Result<(), String>;
}
struct NullVerifier; // enabled=false; verify→1.0; enroll→Ok(()) (no-op)
```

Точка вызова — строго после wake, на аудио срабатывания. v1: `NullVerifier`, verification выкл ⇒
любой голос проходит (нет false-reject пользователя). Реальный эмбеддер — позже, той же
сигнатурой, не трогая wake-слой. Конфиг: вкл/выкл (выкл), порог, путь профиля в `~/.jarvis/`.

## 4. Действие wake (v1)

§16 спеки: wake лишь «дёргает существующий STT-захват», полный голосовой цикл — следующая фаза.
v1-действие (за интерфейсом `WakeAction`, тестируется моками):
1. emit «wake» в панель (видимая реакция);
2. открыть `CaptureSession` с прероллом → фикс-окно (нет VAD в v1) → `SttService::transcribe`;
3. текст → агент через `ClaudeCliHost::run(...)` (тот же вход, что у хоткея).
Любой шаг fail-safe (лог, без паники). Гейт mute/verify — перед открытием захвата.

## 5. Конфиг (`~/.jarvis/settings.json`)

```jsonc
"wake": { "enabled": false, "engine": "openwakeword",
          "model": "<path|hey_jarvis>", "threshold": 0.5, "debounce": 2 },
"verification": { "enabled": false, "threshold": 0.5, "profile": "<path>" },
"stt": { "audioDevice": "...", "mute": false }   // аудио-вход общий с инкр.9
```
`WakeConfig::from_settings` / `VerifyConfig::from_settings` — дефолты-на-дыры, как `SttConfig`.

## 6. Капабилити (доступ для плагина — через гейт)

- **`wakeword.status`** (Read, Trusted): `{enabled, listening, muted, model_present, mic_perm}`.
  Наблюдаемость; безопасно для агента/плагина.
- Включение always-on микрофона **не** капабилити (приватно-критично) — только панель (Settings,
  security-key). Wake-события плагинам — отложено (§9).
- Приватный доступ плагина к речи — через уже существующую `stt.transcribe` (грант на микрофон).

## 7. Панель

Карточка Wake-word: тумблер; тест фразы (сказать «Hey Jarvis» → увидеть срабатывание/скор);
индикатор «слушаю» (из state хаба); жёсткий mute. Раздел Verification — выключен, помечен
«шов, реализация позже», место под enrollment. IPC-команды + emit состояния (как voice/stt).

## 8. Установщик

`install_wakeword`: идемпотентно тянет 3 ONNX (melspectrogram 1.04МБ, embedding_model 1.27МБ,
hey_jarvis_v0.1 1.21МБ) из `github.com/dscripka/openWakeWord/releases/download/v0.5.1/` в
`~/.jarvis/wakeword/`, atomic-write + backup, как silero/stt. `status`: модели на месте, движок,
вкл/выкл, разрешение микрофона. Без сети — компилируется (фича выкл), детект требует файлов.

## 9. Сборка и фичи

- Аудио-фронтенд (cpal/rubato/ring/fan-out/mute) — **всегда в сборке** (лёгкое, офлайн, тестируемо).
- `ort`+openWakeWord — за фичей **`wakeword-ort`** (выкл по умолчанию): `ort` опционален,
  `commit_from_file`, офлайн через бандл dylib (load-dynamic + ORT_DYLIB_PATH) либо
  download-binaries при наличии сети. Дефолтная сборка не требует onnxruntime.
- MSRV: при апгрейде cpal/rubato — поднять `rust-version`. v1 остаётся на cpal 0.15.3/rubato
  0.16.2 (как инкр.9) — без апгрейда, единая версия с инкр.9.

## 10. Тесты (без живого микрофона)

- `Pipeline`: downmix, стрим-ресемпл (48к/32к→16к, ±5%), нарезка ровно 1280, preroll-ring (cap,
  drop-oldest), fan-out (N подписчиков получают одинаковые кадры), **mute → ноль кадров**.
- Wake: `FixtureEngine` срабатывает по маркеру → debounce/warm-up → action вызван;
  порог управляем; стаб инертен (0.0, не будит).
- Verify: `NullVerifier` проходит всегда; точка вызова после wake (гейт пропускает).
- Config: `from_settings` дефолты-на-дыры, мусор→дефолт.
- Capability: `wakeword.status` Read доступна; форма ответа.
- Fail-safe: сбой движка/моделей не роняет демон.

## 11. Не делать (из §16 спеки)

Реальную верификацию (только шов+`NullVerifier`); enrollment-логику; мультипрофили; wake-события
плагинам; полный голосовой цикл; коммерческие движки; трактовать верификацию как security.

## 12. Честная рамка (README)

Верификация — анти-случайное-срабатывание, **не security**. Короткое слово — предел точности.
Always-on приватен: локально, кольцевой буфер, ничего не персистится/не уходит в сеть; индикатор
и жёсткий mute. Wake-word работает офлайн. Полный голосовой цикл — следующая фаза.
