# Дизайн: разговорный голосовой ассистент Jarvis (Haiku-мозг)

Дата: 2026-06-27
Статус: на ревью (ревизия 2 — после мульти-агентного аудита спеки, 39/42 находок подтверждены)
Под-проект 2 из дорожной карты «Jarvis как полноценный голосовой ассистент».
Строится поверх под-проекта 1 (голосовая маршрутизация, модуль `route/`).

## Контекст и место в дорожной карте

Под-проект 1: «Hey Jarvis» → STT → детерминированный роутер → stage-then-send в
сессию, с HUD и индикатором «слышу». Маршрутизация была ЕДИНСТВЕННЫМ исходом.

Под-проект 2: **разговорный ассистент**. Джарвис слушает реплику и сам решает —
это вопрос («сколько времени?», «что у frontend?»), действие над инфраструктурой
(«переключи на opus», «не спи 2 часа») или промпт для сессии. Отвечает **голосом**.
Маршрутизация из п/п-1 становится ОДНИМ из «скилов».

Согласовано на брейншторме:
- **Скоуп — всё**: чтение (sessions/chats/metrics/tasks/limits/time), маршрутизация,
  управление (model/effort/keep-awake/mute/settings).
- **Диалог — полный многоходовый** (память разговора).
- **Модель — только Haiku** через проксированный `claude_bin::run_haiku`.
- **Конец разговора** — тишина ~8-10с ИЛИ стоп-фраза; **барж-ин — акустический,
  через AEC** (конечная цель).
- **Архитектура мозга — подход A**: Rust-оркестратор + структурный single-shot Haiku.

### Вехи (после аудита — обязательное разбиение)

Аудит показал: этапы 1-3 plan-ready на стабильной инфре; 4-6 — research-shaped
(VAD, эхо). Поэтому **три вехи, каждая = свой план реализации**:

- **2a — одноходовый голосовой Q&A** (этапы 1-3): «Hey Jarvis → вопрос/действие →
  голосовой ответ» НА СУЩЕСТВУЮЩЕМ пути захвата (фикс-окно). Поставляемо, plan-ready.
- **2b — многоходовый полудуплекс** (этапы 4-5): VAD-эндпойнтинг + память; пока
  Джарвис говорит — **микрофон замьючен** (полудуплекс), открывается после TTS.
- **2c — акустический барж-ин через AEC** (этап 6): эхо-подавление (macOS
  `VoiceProcessingIO` / Silero-VAD), full-duplex, детект начала речи поверх TTS.
  Research-веха; целевой барж-ин, который заказан.

Этот документ — единая спека всех трёх; планы пишутся по вехам отдельно, начиная с 2a.

## Что уже есть и что НЕТ (выверено по коду ветки voice-routing-rebuilt)

ЕСТЬ (переиспользуем):
- Haiku c проксей: `claude_bin::run_haiku(prompt, timeout) -> Option<String>` —
  хардненный single-shot. Шаблон терпимого парса JSON из ответа Haiku уже есть в
  `daemon.rs` (`ai_toast_summary` парсит `{display,speak}` с fence/prose-толерантностью).
- Под-проект 1: `route::{score, stage, pick, classify, hud}` + `route::SingleFlight`.
  `route::stage::stage` принимает непрозрачный `Box<dyn Send>` держатель — route-как-скил
  переиспользуется чисто.
- Ядра управления вызываются in-process из Rust: `ipc::reply_core`,
  `ipc::set_model_core`/`set_effort_core` (через `set_via_slash` → tmux),
  `Power::cmd(d,"keep-awake",…)`, `Voice::set_mute`.
- Поток кадров `AudioHub` (`WakeTap`/`CaptureSession`, `Arc<[f32]>`, FRAME_LEN=1280 =
  80мс @16к) — источник для VAD: `convo/listen.rs` сам считает поэкадровую энергию.
- TTS Silero: очередь, mute, ducking; примитив прерывания `RodioPlayer::stop()` есть
  (вызывается из `set_mute`).
- `Session`-поля для снапшота (`model.rs`): id/project/task/status/branch/last_prompt/
  tmux_pane/model/effort + счётчики из `daemon.snapshot()`.

НЕТ (надо строить — выявлено аудитом):
- **Нет AEC** (эхо-подавления) нигде в `stt/`+`voice/`. Захват (cpal input) и
  воспроизведение (rodio output) — независимые устройства; TTS течёт в открытый мик.
  → блокер акустического барж-ина (веха 2c); до него — полудуплекс (мьют мика на TTS).
- **`Voice` без публичного `stop()`** и `SpeechQueue` без `clear()`. Нужен
  `Voice::stop()` = `player.stop()` (без mute) + `SpeechQueue::clear()`.
- **TTS fire-and-forget, без сигнала завершения**: `speak_text/test_phrase` ставят
  в очередь и возвращаются; воркер играет позже. Нужен `Voice::speak_blocking(text)`
  с сигналом завершения — иначе цикл начнёт слушать поверх своего голоса.
- **`speak_text` — нотификационный** (дедуп по `kind:title:body`, держит тост);
  повторные ответы («не расслышал», два раза «сколько времени») молча отбрасываются.
  Нужен `Voice::say(text)` — без контент-дедупа, без тоста, `Priority::Done`.
- **`is_mic_silent()`/`audio_state` — НЕ источник VAD**: это грубый ~10с watchdog
  (TCC/тишина), не годен для 0.8с-эндпойнтинга. VAD считает RMS сам по кадрам.
- **Нет consent для CONTROL**: in-process путь (`via_gate_panel`+`Consumer::panel`+
  AutoApprove) ОБХОДИТ подтверждение; `keep-awake`/`mute` вообще не капабилити (нет
  гейта). `set_model_core`/`set_effort_core` ПАСТЯТ `/model {x}`/`/effort {x}` в пану
  БЕЗ валидации. → нужен явный consent + валидация (см. §Безопасность).
- **Wake-детектор не паузится во время разговора** → повторный «Hey Jarvis»
  (или срабатывание на свой TTS) плодит второй цикл. Нужен conversation-level lock.

## Архитектура (подход A): Rust-оркестратор + структурный Haiku

Мозг — НЕ MCP-агент, а **планировщик на один ход**: Rust собирает контекст, делает
ОДИН `run_haiku` со структурным выводом, парсит план, исполняет. Играет на сильную
сторону Haiku (одно структурное решение), Rust держит безопасность/HUD/TTS/память/
латентность, всё юнит-тестируемо.

### Снапшот мира (ключ к «один вызов на ход»)
`build_snapshot(daemon)` всегда в промпте: время/дата; живые сессии
`{id[..8], project, task, status, branch, last_prompt(сжат), in_tmux}`; счётчики
ждут/работают; флаг лимита; mute; keep-awake. → Haiku отвечает на большинство
read-вопросов одним вызовом (`speak`), без отдельного скила. Глубокие данные
(полный чат, детальные метрики) — через `action` (read-скил), Rust достаёт, при
`need_followup` — 2-й вызов фразит. Бюджет ≤2 вызова Haiku/ход.

## Формы данных (пин для TDD)

```rust
// План — структурный вывод Haiku за ход
struct Plan {
    speak: String,             // что озвучить (может быть пусто, если только action)
    action: Option<Action>,    // ≤1 действие за ход
    end: bool,                  // Haiku распознал конец («спасибо/хватит»)
    need_followup: bool,        // после read-результата нужен 2-й вызов сфразить
}
struct Action { skill: String, args: serde_json::Value }

// Исход исполнения скила
enum SkillOutcome {
    Data(serde_json::Value),       // read → данные для (опц.) 2-го вызова
    Staged,                        // route/control → ушло в окно отмены/подтверждение
    Rejected(String),              // нелистовой скил / провал валидации → переспрос
}
```
Снапшот сериализуется компактно (строки `project · task · status`, не сырой JSON).
Память хода: `{user: String, assistant: String, action_result: Option<String>}`,
где `action_result` — КОРОТКАЯ структурная сводка (skill + санированные args +
код исхода), НИКОГДА не сырой текст `chats.read`/документов (см. §Безопасность).

## Меню скилов и валидация аргументов

Чтение (free, без consent):
- `sessions_status` · `session_detail{id}` · `session_chat{id}`(chats.read) ·
  `metrics{scope}` · `tasks{id}` · `limits` · `time`(Rust, без модели).

Маршрутизация (сайд-эффект → окно отмены/пикер из п/п-1):
- `route{prompt}` — БЕЗ `session_hint`: `route::score` ранжирует цель из полей
  сессий (как сейчас), неоднозначно → пикер. (Сверено: `route_transcript`/`score::rank`
  берут только транскрипт; параметра hint нет — не выдумываем.)

Управление (сайд-эффект → ПОЗИТИВНОЕ подтверждение, не пассивное окно):
- `set_model{id, model}` · `set_effort{id, level}` · `keep_awake{minutes|off}` ·
  `mute{on|off}`.

**Валидация (fail-closed, ДО исполнения)** — `convo/skills.rs`:
- `model` ∈ фикс-allowlist (opus/sonnet/haiku/fable/…); `level` ∈ {low,medium,high,
  xhigh,max} (из `daemon.effort_levels`); `minutes` ∈ разумный диапазон;
  `id` обязан резолвиться через `d.session(id)`.
- Любое значение с пробелами/переводами строки/control-символами → отказ (защита от
  пасты slash-команды в пану). Промах → `Rejected` → переспрос ровно один раз.
- Аллоулист вынести в общий модуль — им же закрыть существующий IPC-чокпойнт
  `set_model_core`/`set_effort_core` (панель сейчас тоже шлёт невалидированно).

## Модель безопасности (микрофон недоверенный, многоход)

- **Reads — свободно.** Сайд-эффекты — только через до-исполнительный consent.
- **CONTROL → позитивное подтверждение** (НЕ пассивный 5с-таймер): прогоняем
  control-скилы через существующий гейт как `Consumer::agent()` (`ConfirmPolicy::Always`
  → `PanelConfirmer` с точным описанием эффекта и oneshot-решением), ИЛИ эквивалентную
  confirm-карточку. Пассивное 5с-окно `route::stage` оставляем ТОЛЬКО для route/reply
  (там юзер только что продиктовал точный текст). Джарвис ОЗВУЧИВАЕТ эффект при
  подтверждении (TTS), чтобы согласие было слышно, когда HUD не виден.
- **`keep-awake` и `mute` обернуть**: зарегистрировать как `RiskClass::Control`
  капабилити (делегируют в `Power::cmd`/`set_mute`) ЛИБО провести через тот же
  confirm. **`mute{on}` голосом без подтверждения запрещён** — он глушит звуковой
  аудит-след (можно скрыть последующие подтверждения).
- **In-process диспатч обходит капабилити-гейт** — это осознанно (как `reply_core` в
  п/п-1); поэтому границей согласия служит наш convo-confirm/stage, а для control —
  явный confirm выше. Спека фиксирует это прямо.
- **Гигиена памяти**: в память кладём только user/assistant/короткую сводку исхода;
  сырой untrusted-текст (`chats.read`, документы, `last_prompt`) в память НЕ попадает
  — иначе инъекция из одного хода переезжает во все следующие промпты.
- **Taint-трекинг**: если контекст хода включал untrusted read-вывод
  (`CallOutput.provenance == Untrusted`), любой сайд-эффект этого/следующего хода идёт
  ТОЛЬКО через явный confirm (не окно). In-prompt маркер «данные, не команды» —
  defense-in-depth, не основной контроль.
- Барж-ин/прерывание безопасны (стоп TTS).

## TTS-ответ (новые методы `Voice`)

- `Voice::say(text)` — план-речь: enqueue без контент-дедупа (или dedup_key =
  счётчик хода), `Priority::Done`, `toast_id: None`. НЕ `speak_text`.
- `Voice::speak_blocking(text) -> SpeakOutcome` — enqueue с oneshot/Condvar-сигналом,
  блокирует вызывающий convo-поток до конца `play_blocking` ИЛИ прерывания; возвращает
  `Done|Interrupted`. Это даёт цикл SPEAK→LISTEN (полудуплекс), переиспользуя воркер,
  очередь, ducking, mute. (Учесть приоритетную преемпцию: NeedHuman-нотификация может
  вызвать `player.stop()` посреди реплики — трактуем как `Interrupted`, не молчаливый обрыв.)
- `Voice::stop()` = `player.stop()` (без mute/force_unduck) + `SpeechQueue::clear()`
  (добавить `clear()`); иначе воркер заговорит следующую из очереди.

## Многоходовость (веха 2b, полудуплекс)

- **Полудуплекс**: пока Джарвис говорит — слушание выключено (мик-мьют/игнор VAD);
  окно слушания открывается ПОСЛЕ `speak_blocking` вернул `Done`. Это убирает
  само-срабатывание VAD до появления AEC (2c).
- **VAD-эндпойнтинг** (`convo/listen.rs`): НОВЫЙ потоковый WakeTap-консьюмер
  (`recv_timeout`, 80мс-кадры) + чистая RMS-функция на кадр (юнит-тест). Автомат:
  старт по энергии → запись → трейлинг-тишина ~0.8с (≈10 кадров) → конец; общий
  таймаут ожидания старта 8-10с → конец разговора. Порог **адаптивный** (калибровка
  шумового пола первых N кадров) — нет AGC в пайплайне.
- **Память** (`convo/memory.rs`): скользящее окно последних N ходов, ограничение по
  токенам; формат как в §Формы данных (без сырого untrusted).
- **Конец**: тишина-таймаут ИЛИ `plan.end` (+ локальный детект стоп-фразы); короткое
  голосовое прощание.
- **Подавление wake во время разговора**: детектор ПРОДОЛЖАЕТ скорить кадры
  (`run_loop` не паузится), поэтому повторный вход блокируем на УРОВНЕ ДЕЙСТВИЯ:
  conversation-level флаг (проверяется в `on_wake` до `SingleFlight::try_enter`),
  держится весь разговор, снимается на END. НЕ переиспользовать per-turn
  `route::stage` Box-held (он дропается после 5с-окна каждого хода) — route-скил
  внутри разговора берёт свой stage-токен, отдельный от conversation-lock.

## Акустический барж-ин (веха 2c — целевой, research)

- **AEC обязателен** (без него энергетический детектор слышит сам Джарвиса).
  Рекоменд. путь на macOS: вход через **`VoiceProcessingIO` audio unit**
  (`kAudioUnitSubType_VoiceProcessingIO`, встроенный AEC) ВМЕСТО дефолтного cpal-входа
  — заметная нативная работа (CoreAudio), либо софт-AEC (webrtc-audio-processing /
  speexdsp) с подачей reference-сигнала TTS. Альтернатива/дополнение — Silero-VAD.
- При наличии AEC: full-duplex — слушаем поверх TTS, детект начала речи (онсет) →
  `Voice::stop()` → захват реплики. Полудуплекс из 2b остаётся фолбэком, если AEC
  недоступен/выключен.
- **Прероборные риски** (исследовать в начале 2c): качество AEC на встроенных
  динамиках; интеграция VoiceProcessingIO с текущим `AudioHub`/cpal; задержка.

## Латентность (ход, Haiku)
VAD/захват + STT (~0.5-2с) + Haiku-план (хардненный, ~1-3с) + TTS до первого звука
(~0.5-1.5с) ≈ 3-7с/ход с HUD «Думаю…». Снапшот в промпте убирает лишние round-trip’ы.

## Этапы реализации (по вехам, TDD)

**2a (одноходовый Q&A):**
1. `snapshot.rs` + `plan.rs` (чистые) — мозг (формы данных выше).
2. `skills.rs` — диспатч + валидация: reads → данные; route → `route::stage`; control →
   confirm; нелистовой/невалидный → `Rejected`.
3. `reply.rs` — `Voice::say`/`speak_blocking` + HUD-фазы; одноходовый «вопрос →
   голосовой ответ» поверх текущего фикс-окна. **Поставляемо.**

**2b (многоход, полудуплекс):**
4. `listen.rs` — WakeTap-консьюмер + RMS + адаптивный порог + автомат endpointing.
5. `convo/mod.rs` + `memory.rs` — цикл, память, условия конца, conversation-lock,
   подавление wake-действия.

**2c (акустический барж-ин):**
6. AEC (VoiceProcessingIO/Silero-VAD) + онсет-детект поверх TTS + `Voice::stop()` →
   full-duplex. Research-веха.

Каждый этап — рабочий коммит. После 2a — голосовой Q&A; после 2b — полноценный
полудуплексный разговор; после 2c — перебивание голосом.

## Тестирование (TDD)
- `plan.rs` (чистая): промпт содержит снапшот/меню/память/untrusted-маркер; `parse_plan`
  терпим (fence/конверт/мусор → безопасный фолбэк-переспрос).
- `snapshot.rs` (чистая): корректная компактная сводка + счётчики/лимит/mute.
- `skills.rs`: read→`Data`; route/control→`Staged` (мок ядра); нелистовой→`Rejected`;
  **валидация fail-closed**: плохая модель/effort/минуты/несуществующий id → `Rejected`;
  значение с whitespace/control → `Rejected`.
- RMS-функция + `listen.rs`-автомат на синтетических кадрах (старт/конец/таймаут).
- `memory.rs`: ограничение окна; **сырой untrusted-текст НЕ хранится** (инвариант).
- Оркестратор: последовательность ходов, конец, read→re-plan (фейк Haiku+ядро).
- Безопасность: сайд-эффект недостижим без consent; control → позитивный confirm, не
  окно; **cross-turn**: инъекция в `chats.read` хода N не авто-исполняет действие в
  N+1, идёт в confirm, и её сырой текст отсутствует в промпте N+1; `mute{on}` голосом
  требует confirm.
