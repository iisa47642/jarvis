# Авто-подхват живых tmux-сессий при рестарте демона

**Дата:** 2026-06-18
**Статус:** одобрено, реализуется

## Проблема

После перезапуска демона jarvis он не подхватывает Claude-сессии, которые
по-прежнему живут в tmux-сервере `-L jarvis`. Причина: на старте демон делает
только `restore_state()` — читает `~/.jarvis/state.json` — и **никогда не
сканирует живой tmux**. Следствия:

- сессии старше 24ч выкидываются cutoff'ом в `restore_state`;
- сессия, поднятая **пока демон лежал**, в `state.json` не попадает → невидима;
- битый/пустой `state.json` → все живые паны осиротели;
- фоновый `sweep_sessions()` (раз в 30с) умеет только **удалять** мёртвые паны,
  живые осиротевшие обратно не подбирает.

Итог: в tmux Клоды живы и ждут ввода, а демон про них не знает.

## Решение (вариант «лёгкий адопт»)

Превратить одностороннюю чистку `sweep_sessions()` в двунаправленную
`reconcile_sessions()`: помимо удаления мёртвых пан — **подхватывать живые паны,
не закреплённые ни за одной сессией**, заводя «провизорную» сессию. Запуск — там
же, где сейчас sweep: на старте и в 30-секундном цикле (значит ловятся и сессии,
поднятые во время простоя демона).

### Ключевой переиспользуемый механизм

В `reduce()` уже есть инвариант «одна пана — одна сессия»: `evict_pane()`
выселяет любую другую сессию, державшую пану, когда по этой пане приходит хук.
Провизорная сессия держит `tmux_pane`, поэтому **сама выселится**, когда из её
паны прилетит первый настоящий хук — настоящая сессия создаётся с реальным
`session_id` и полными данными. Никакой ручной merge-логики и дублей.

## Изменения

### 1. `model.rs` — поле-маркер
Добавить в `Session` поле `adopted: bool` (skip при `false`, как остальные).

### 2. `tmux.rs` — богатый список пан
Добавить `PaneInfo { pane_id, session_name, cwd, pid }` и `list_panes_meta()` с
форматом `#{pane_id}\t#{session_name}\t#{pane_current_path}\t#{pane_pid}`.
Сохранить семантику арм как у `list_panes`: `Ok(Some)` — успех, `Ok(None)` —
tmux не найден (не трогаем реестр), `Err(())` — ошибка/пустой сервер.

### 3. `daemon.rs` — reconcile вместо sweep
- Переименовать `sweep_sessions` → `reconcile_sessions`, сохранив существующее
  удаление мёртвых пан и заморозку досок.
- Добавить шаг адопта: для каждой живой паны, чей `pane_id` не встречается ни в
  одном `tmux_pane` реестра, создать провизорную сессию:
  - ключ `pane:<pane_id>`;
  - `adopted = true`, `status = Idle` (кнопка «Продолжить» только для
    `waiting/limit` → скрыта; в сортировке внизу; уведомление/звук не дёргаются,
    т.к. адопт идёт тихо, минуя `notify()`);
  - `tmux_pane`, `tmux_name`, `cwd`/`project`, `pid` из `PaneInfo`;
  - `detail = "восстановлена"`.
- Адоптить только при успешном листинге (`Ok(Some)`); при `Ok(None)` реестр не
  трогать; при `Err(())` — как сейчас (удаление по пустому набору, адопт ноль).

### 4. `write_state_now` — не персистим провизорные
Фильтровать `adopted` при записи `state.json` — они всегда заново выводятся из
живого tmux, чтобы не плодить протухшие записи и не конфликтовать с
`restore_state`.

### 5. `main.rs` — call-site
`dd.sweep_sessions()` → `dd.reconcile_sessions()`.

### 6. `renderer.js` — UI
При `adopted: true` — приглушённая плашка «восстановлена». Кнопка «Продолжить»
уже скрыта статусом `Idle`, спец-логики не нужно.

## Крайние случаи

- Все паны сервера `-L jarvis` — агенты (шим только так их создаёт) → адоптим
  все паны без фильтра по команде.
- Сессия, пережившая 24ч-cutoff, но живая в tmux: восстановится как провизорная
  (без старой истории/доски, но видимая и управляемая) — приемлемо для варианта 1.
- Сессия с непустой доской: при смерти паны существующая логика её замораживает
  (status=Done, «сессия остановлена»), адопт сюда не вмешивается.

## Проверка

- `cargo build` / `cargo test` (юнит на адопт + не-персист + эвикт провизорной).
- Ручной сценарий: поднять Клода в `-L jarvis`, прибить демон, перезапустить →
  сессия появляется как «восстановлена»; послать хук → апгрейд до настоящей.
