# Спайк 2c: акустический барж-ин (AEC) — спека + честная граница

Дата: 2026-06-27
Статус: спайк (вслепую, без микрофона — части помечены как непроверяемые)
Веха 2c под-проекта 2. Строится поверх 2a/2b (разговорный ассистент, полудуплекс).

## Цель
Перебивать Джарвиса голосом ПОСРЕДИ его речи (full-duplex): пока идёт TTS, слушать
микрофон, по началу речи (онсет) — оборвать озвучку (`voice::stop`) и перейти к
захвату новой реплики. Заказано пользователем именно как АКУСТИЧЕСКИЙ барж-ин.

## Несущая проблема (из аудита спеки и реализации)
В стеке НЕТ эхо-подавления (AEC): захват (cpal default_input) и воспроизведение
(rodio default_output) — независимые устройства. Пока TTS играет в динамики, его
звук попадает в открытый микрофон. Энергетический онсет-детектор примет голос
Джарвиса за речь юзера → мгновенный ложный барж-ин (оборвёт сам себя). Поэтому
**full-duplex барж-ин невозможен без AEC** (или наушников).

## Что РЕАЛЬНО делаем в спайке (тестируемо, без микрофона)
1. **`voice::stop()` + `SpeechQueue::clear()`** — публичный примитив прерывания:
   оборвать текущую озвучку (`player.stop()`) И очистить очередь (иначе воркер
   заговорит следующую). Без mute/force_unduck. Юнит-тест на `clear`.
2. **`EchoCanceller` trait + `NoopAec`** — шов абстракции AEC: `process(mic_frame,
   ref_frame) -> frame`. Noop = passthrough (текущее поведение). Реальный backend
   (macOS VoiceProcessingIO / webrtc-audio-processing) подключается сюда позже.
3. **Onset-детектор для барж-ина** (`convo/bargein.rs`, чистый): автомат «во время
   TTS появилась речь выше порога N кадров подряд» → сигнал прерывания. Порог —
   над уровнем эха (когда AEC реальный) ИЛИ грубо завышенный (паллиатив). Юнит-тест.
4. **Flag-gated проводка** (`bargeIn` в настройках, ДЕФОЛТ OFF): когда включено,
   `speak_blocking` параллельно гоняет onset-детектор на WakeTap; онсет →
   `voice::stop()` → выход из speak_blocking с флагом «прервано» → цикл сразу
   слушает. Когда выключено (дефолт) — поведение 2b (полудуплекс) без изменений.

## Что НЕ делаем вслепую (граница честности)
**Нативный `VoiceProcessingIO` AudioUnit** (встроенный AEC на macOS) — замена
cpal-входа на CoreAudio AudioUnit c kAudioUnitSubType_VoiceProcessingIO,
render-callbacks, unsafe FFI. Это сотни строк нативного кода, корректность которого
определяется акустикой устройства и проверяется ТОЛЬКО на железе. Писать его вслепую
= почти гарантированный неработающий/некомпилирующийся код + переделка. Оставляем
чёткий шов (`EchoCanceller`) и включаем барж-ин флагом лишь когда backend готов и
оттюнен на микрофоне. Дефолт OFF → 2c в спайке безопасен (не ломает 2a/2b).

## Альтернативы AEC (для реализации backend позже)
- **macOS VoiceProcessingIO** (рекоменд.): встроенный AEC/AGC/NS в OS; нужен
  AudioUnit-вход вместо cpal default. Лучшее качество, нативно, без новых крейтов.
- **webrtc-audio-processing** (крейт): софт-AEC, нужен reference-сигнал (то, что
  играем) синхронно с mic. Кросс-платформенно, но новая зависимость + синхронизация.
- **Silero-VAD** вместо энергии: не решает эхо (распознает речь Джарвиса как речь),
  но устойчивее к шуму; полезно В ПАРЕ с AEC, не вместо.

## Тестирование (спайк)
- `SpeechQueue::clear()` дренирует очередь; `voice::stop` не мьютит.
- `bargein` onset-автомат: N кадров выше порога подряд → Triggered; ниже — сброс;
  эхо-уровень (ниже порога) НЕ триггерит (с реальным AEC).
- `NoopAec::process` = passthrough (длина/данные не меняются).

## Вне scope
Сам нативный AEC-backend (мик-зависимый), тюнинг порогов на железе, полноценный
Silero-VAD. Включение `bargeIn` по умолчанию — только после мик-валидации backend.
