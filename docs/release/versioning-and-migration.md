# Политика версий, обновлений и миграций

Цель: обновления Jarvis **не ломают** существующий функционал и **не сносят**
настройки/данные пользователя. Документ — нормативный; релиз делается по скиллу
`release-jarvis`.

## 1. Версионирование (semver)

`MAJOR.MINOR.PATCH`:
- **PATCH** (`0.3.0 → 0.3.1`) — багфиксы, без изменения поведения и схем.
- **MINOR** (`0.3.0 → 0.4.0`) — новые фичи, обратно совместимо; схемы только
  расширяются (новые поля с дефолтами).
- **MAJOR** (`0.x → 1.0`) — ломающие изменения (смысл полей, удаление команд,
  несовместимый формат данных). Требует миграции и явного решения.

Версия живёт в 4 местах и должна совпадать: `package.json`,
`src-tauri/tauri.conf.json`, `src-tauri/Cargo.toml`, `src-tauri/Cargo.lock`.

## 2. Данные пользователя переживают обновление

Все пользовательские данные — в `~/.jarvis/` (или `$JARVIS_DIR`), **вне** бандла
`.app`. Апдейт заменяет только `.app` → данные нетронуты.

Там лежат: `settings.json`, `state.json`, `history.json`, `usage.json`,
`voice-history.json`, `convo-memory.json`, `tokens.json`, `metrics.jsonl`, логи,
и подкаталоги моделей (`silero/`, `stt/`, `stt-mlx/`, `wakeword/`), `bin/`,
`shims/`, `codex-sdk/` и др.

**Правила:**
- НИКОГДА не хранить пользовательские данные/настройки внутри `.app`.
- НИКОГДА не удалять `~/.jarvis` при обновлении (только `jarvis-setup uninstall`
  по явному запросу пользователя).
- Секреты (`settings.json`, `tokens.json`) — права `0o600`, не логировать.

## 3. Совместимость схем (как не сломать)

### settings.json
Загрузка мержит дефолты с диском → **добавление полей безопасно**. Правила:
- Новое поле — всегда с `#[serde(default)]` (или дефолтом в merge), чтобы старый
  файл без него читался.
- НЕ менять смысл существующего поля. Нужно другое поведение — заведи новое поле.
- Новое поведение, способное сломать привычный UX, — за **дефолтным флагом**
  (выключено/как раньше по умолчанию).
- Удаление/переименование/реструктуризация поля — только через миграцию (ниже).

### state.json / history.json / прочее
- Парс-ошибка → тихий скип (данные игнорируются, не падаем). Уже реализовано.
- Ломающий формат — бамп формата + миграция или осознанный сброс именно этого
  файла (не всего `~/.jarvis`).

## 4. Механизм миграции

Когда обратной совместимости мало (переименование/смена смысла/реструктуризация):

1. В `settings.json` есть поле `schemaVersion` (целое, отсутствие = 0).
2. При загрузке `settings.rs` прогоняет миграции по возрастанию версии:
   `migrate_v0_to_v1`, `migrate_v1_to_v2`, … — каждая чисто преобразует JSON.
3. После миграций `schemaVersion` поднимается до текущего и файл переписывается.
4. Каждая миграция покрыта юнит-тестом (старый JSON → ожидаемый новый).

Принципы: миграции **идемпотентны** и **только вперёд**; downgrade не
поддерживаем (старый бинарь просто проигнорит незнакомые поля). Перед рискованной
миграцией — бэкап `settings.json` → `settings.bak.json`.

## 5. Авто-обновление

- Подписанный апдейт (minisign): `plugins.updater.pubkey` ↔ секрет
  `TAURI_SIGNING_PRIVATE_KEY`. Endpoint → `latest.json` последнего релиза.
- Клиент проверяет апдейт на старте; скачивает и ставит свежую версию.
- Пользователь на версии без корректного endpoint/pubkey (напр. ранний 0.2.0 с
  placeholder) обновляется **один раз вручную**; дальше — автоматически.
- Поведение установки задаётся настройкой (см. `settings.json`, поведение
  апдейтера): по умолчанию — не сносить рабочий процесс врасплох.

## 6. Откат

- Релизы неизменяемы; откат = выпустить новый PATCH с фиксом (вперёд), а не
  переписывать опубликованный тег.
- Если ломающая миграция уже уехала — следующий релиз чинит данные ещё одной
  миграцией, не требуя ручных действий пользователя.

## 7. Чек-лист «не сломать» перед релизом

- [ ] Новые поля настроек — с дефолтами; смысл старых не изменён
- [ ] Ломающие изменения схемы — есть миграция + тест + бамп `schemaVersion`
- [ ] Ничего пользовательского не пишется внутрь `.app`
- [ ] `~/.jarvis` не трогается обновлением
- [ ] Прогон апдейта с предыдущей версии (по возможности) — настройки на месте
